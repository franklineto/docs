Arquitetura 3:
HA residencial com replicação bidirecional

by Franklin

Como construir uma infraestrutura de alta disponibilidade entre duas residências usando Cloudflare Tunnel para roteamento com failover automático, bancos locais em cada site, e um hub de replicação na nuvem para manter tudo sincronizado. Quatro estratégias de replicação são avaliadas em detalhe — escolha a que melhor se encaixa no seu stack.

01 — FundaçãoVisão geral da arquitetura

A Arquitetura 3 divide o sistema em quatro camadas independentes, cada uma resolvendo um problema específico de alta disponibilidade. O princípio central é que nenhuma camada sabe sobre falhas das outras — cada uma tem seu próprio mecanismo de recuperação, o que aumenta a resiliência total.

As quatro camadas

  1. Roteamento. O Cloudflare Load Balancer detecta qual site está online e direciona o tráfego automaticamente.
  2. Aplicação. Containers Docker idênticos rodam em ambas as residências, implantados a partir do mesmo repositório Git.
  3. Arquivos. Uploads de usuários e assets mutáveis são sincronizados peer-to-peer via Syncthing.
  4. Banco de dados. Cada site tem seu banco local; as mudanças são propagadas via um mecanismo de replicação (aqui estão as 4 sub-opções).

Topologia física

Duas residências, cada uma com um servidor local (um mini-PC, Raspberry Pi 5, NUC ou similar). Cada servidor roda a stack completa: aplicação + banco + Syncthing + cloudflared. A Cloudflare funciona como ponto único de entrada público — nenhuma das residências precisa de IP público.

Princípio de design Trate cada site como se fosse o único — ambos devem ser capazes de operar independentemente por horas ou dias. A replicação é um mecanismo de convergência eventual, não um requisito para operar.

02 — PreparaçãoRequisitos e premissas

Hardware mínimo (por residência)

  • CPU: 4 cores (ARM ou x86) — Raspberry Pi 5, mini-PC ou NUC resolvem.
  • RAM: 8 GB (16 GB recomendado se usar Postgres com carga real).
  • Storage: SSD NVMe de pelo menos 256 GB — evite HDD para o banco.
  • Rede: 100 Mbps symmetric idealmente; 20 Mbps de upload é o mínimo funcional.
  • UPS (nobreak): 600 VA mínimo para dar tempo de desligamento limpo.

Software

  • Ubuntu Server 24.04 LTS ou Debian 12 (os comandos do tutorial assumem Debian/Ubuntu).
  • Docker Engine + Docker Compose v2.
  • Git (para deploy da aplicação).
  • cloudflared instalado e autenticado.

Conta e domínio

  • Domínio registrado e já gerenciado pelo Cloudflare (nameservers apontando para eles).
  • Plano Cloudflare: o Load Balancer é um add-on pago (~US$ 5/mês no plano básico). O Tunnel em si é gratuito.
  • Opcional: uma conta em provedor de banco em nuvem (Neon, Supabase, CouchDB Cloud, Turso) dependendo da opção escolhida.
Atenção As duas residências devem estar em redes elétricas e provedores de internet diferentes. Se ambas compartilham o mesmo fornecedor de energia ou o mesmo ISP regional, a redundância é ilusória — uma única falha derruba tudo.

03 — Camada de RoteamentoCloudflare Load Balancer com dois tunnels

Aqui configuramos a peça que decide para qual residência mandar cada request. Vamos criar dois tunnels Cloudflare (um por residência) e um Load Balancer com health checks que faz o failover automaticamente.

Residência 1 — tunnel primário

Instalar e criar o tunnel da Residência 1

Em cada residência, instale o cloudflared e crie um tunnel único:

# Instalar cloudflared (Debian/Ubuntu)
curl -L --output cloudflared.deb \
  https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb
sudo dpkg -i cloudflared.deb

# Autenticar com sua conta Cloudflare
cloudflared tunnel login

# Criar o tunnel desta residência
cloudflared tunnel create casa-01
# Anote o UUID retornado

Residência 1

Configurar o tunnel para apontar para a app local
# /etc/cloudflared/config.yml
tunnel: <UUID-do-tunnel-casa-01>
credentials-file: /etc/cloudflared/<UUID>.json

ingress:
  - hostname: casa-01.seudominio.com.br
    service: http://localhost:8080
  - service: http_status:404

O hostname casa-01.seudominio.com.br é interno — não é para os usuários acessarem. É apenas o endpoint que o Load Balancer vai usar como "origin".

Residência 1

Subir o tunnel como serviço
sudo cloudflared service install
sudo systemctl enable --now cloudflared
sudo systemctl status cloudflared

# Criar a rota DNS para o hostname interno
cloudflared tunnel route dns casa-01 casa-01.seudominio.com.br

Residência 2

Repetir para a segunda residência

Mesmo procedimento, mas com os nomes trocados: cloudflared tunnel create casa-02 e hostname casa-02.seudominio.com.br. A app local escuta na mesma porta (8080), mas cada tunnel aponta para o servidor local da sua casa.

Dashboard Cloudflare

Criar os Origin Pools

No dashboard da Cloudflare, vá em Traffic → Load Balancing → Pools e crie dois:

  • pool-casa-01 — endpoint: casa-01.seudominio.com.br
  • pool-casa-02 — endpoint: casa-02.seudominio.com.br

Dashboard Cloudflare

Criar o Health Check

Em Load Balancing → Monitors, crie um monitor HTTP apontando para o endpoint /health da sua app (você precisa implementar esse endpoint — um simples 200 OK se o banco responder, 503 caso contrário).

  • Intervalo: 30 segundos
  • Retries: 2
  • Timeout: 5 segundos
  • Resposta esperada: HTTP 200

Dashboard Cloudflare

Criar o Load Balancer em modo Failover

Em Load Balancing → Load Balancers, crie um novo apontando para o hostname público real (ex: app.seudominio.com.br):

  • Traffic Steering: Off (Failover)
  • Pool ordering: pool-casa-01 primeiro, pool-casa-02 como fallback
  • Proxy: habilitado (laranja)
  • Anexe o monitor criado no passo anterior aos dois pools

Validação

Testar o failover
# Acessar pelo domínio público e verificar qual tunnel respondeu
curl -s https://app.seudominio.com.br/debug/hostname
# Deve retornar "casa-01"

# Desligar a app na casa-01 (ou parar o tunnel)
sudo systemctl stop cloudflared

# Aguardar ~60s (2 retries × 30s) e testar novamente
curl -s https://app.seudominio.com.br/debug/hostname
# Deve retornar "casa-02"
Endpoint /health bem feito O endpoint /health deve verificar não só se a app está viva, mas se o banco local responde e se a replicação não está atrasada além de um threshold. Isso evita servir tráfego de um site "zumbi" (app online, dados velhos).

04 — Camada de AplicaçãoDocker + Git em ambos os sites

A aplicação precisa ser idêntica nas duas residências. A abordagem padrão é empacotar tudo como imagem Docker e usar Git como fonte da verdade para o código.

Estrutura do repositório

meu-app/
├── Dockerfile
├── docker-compose.yml
├── .env.example
├── src/
└── scripts/
    ├── deploy.sh
    └── healthcheck.sh

Docker Compose de cada site

# docker-compose.yml
services:
  app:
    image: meu-app:latest
    restart: unless-stopped
    ports:
      - "8080:8080"
    environment:
      - SITE_ID=${SITE_ID}  # "casa-01" ou "casa-02"
      - DB_HOST=db
      - DB_PORT=5432
    depends_on: [db]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  db:
    image: postgres:16
    restart: unless-stopped
    volumes:
      - ./pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
      POSTGRES_DB: appdb
    secrets: [db_password]

secrets:
  db_password:
    file: ./secrets/db_password.txt

Deploy automatizado

Um script simples que roda num webhook do GitHub (ou cron a cada 5 minutos) puxa mudanças e recria os containers se houver atualização: 

#!/usr/bin/env bash
# scripts/deploy.sh
set -euo pipefail

cd /opt/meu-app
git fetch origin main

LOCAL=$(git rev-parse HEAD)
REMOTE=$(git rev-parse origin/main)

if [ "$LOCAL" != "$REMOTE" ]; then
  echo "Atualizacao detectada, fazendo deploy..."
  git reset --hard origin/main
  docker compose build
  docker compose up -d
  docker image prune -f
fi

05 — Camada de ArquivosSyncthing para arquivos mutáveis

Uploads de usuários (imagens, PDFs, anexos) e qualquer arquivo que a aplicação escreva em disco precisam ser replicados. O Syncthing é ideal: peer-to-peer, gratuito, sem dependência de nuvem, resolve conflitos por timestamp e suporta versionamento.

Ambos os sites

Instalar o Syncthing
sudo apt install syncthing
sudo systemctl enable --now syncthing@<seu-usuario>

Ambos os sites

Acessar a interface web e trocar os Device IDs

A UI roda em http://localhost:8384. Em cada site, copie o Device ID da interface e adicione o ID do outro site como device remoto. Aceite a conexão nos dois lados.

Ambos os sites

Compartilhar a pasta de uploads

Adicione a pasta /opt/meu-app/uploads como pasta compartilhada, habilite "Send & Receive" nos dois lados e configure versionamento tipo "Staggered" (mantém versões antigas por 30 dias).

Conflitos de arquivo Se os dois sites receberem uploads diferentes com o mesmo nome de arquivo durante uma partição de rede, o Syncthing cria arquivos com sufixo .sync-conflict-<data>. Implemente nomes de arquivo únicos na aplicação (use UUIDs) para evitar isso.

06 — Camada de BancoQuatro opções de replicação

Esta é a decisão arquitetural mais importante — cada opção tem um perfil de trade-offs diferente. Abaixo, cada sub-opção tem instruções de configuração, prós, contras e um veredito sobre quando escolher cada uma.

Opção 3a — PostgreSQL Logical Replication

RELACIONAL · SQL

O PostgreSQL tem replicação lógica nativa desde a versão 10. A ideia: o site ativo publica as mudanças, o site standby e um banco em nuvem (ex: Neon, Supabase) se inscrevem nessas mudanças. Em caso de failover, você promove o standby e reconfigura a direção da replicação.

Passo a passo

# postgresql.conf (em ambos os sites)
wal_level = logical
max_wal_senders = 10
max_replication_slots = 10
wal_keep_size = 1GB
-- No site primário (casa-01)
CREATE PUBLICATION pub_appdb FOR ALL TABLES;

-- No site standby (casa-02) e no banco em nuvem
CREATE SUBSCRIPTION sub_from_casa01
  CONNECTION 'host=casa-01.internal port=5432 dbname=appdb user=replicator password=xxx'
  PUBLICATION pub_appdb
  WITH (copy_data = true, create_slot = true);

Rede entre os sites

O standby precisa alcançar a porta 5432 do primário. Como as duas residências não têm IP público, use Cloudflare Tunnel com acesso TCP privado (WARP client no standby) ou uma VPN mesh como Tailscale — essa é de longe a solução mais simples. 

# Tailscale em ambos os sites
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
# Agora cada site tem um IP 100.x.x.x estável acessível pelo outro

Failover manual

-- Quando casa-01 cair, promover casa-02 a primario:
ALTER SUBSCRIPTION sub_from_casa01 DISABLE;
ALTER SUBSCRIPTION sub_from_casa01 SET (slot_name = NONE);
DROP SUBSCRIPTION sub_from_casa01;
CREATE PUBLICATION pub_appdb FOR ALL TABLES;
-- Quando casa-01 voltar, ela se inscreve em casa-02
Vantagens
  • Nativo do PostgreSQL — zero dependências extras.
  • Replicação granular por tabela/schema.
  • Funciona entre versões diferentes do PG.
  • Latência baixíssima nas operações locais.
  • Ferramentas maduras para automação (Patroni, repmgr).
Desvantagens
  • Failover automático exige Patroni ou similar — não trivial.
  • Não replica DDL (ALTER TABLE precisa rodar manualmente nos dois).
  • Conflitos de escrita simultânea exigem lógica manual.
  • Slots de replicação podem encher o disco se o standby ficar offline muito tempo.
Escolha se… você já usa Postgres, sua aplicação depende de features relacionais (JOINs complexos, transações, constraints) e você está disposto a investir tempo configurando Patroni para failover automático. É a opção mais "profissional" e a que se parece mais com o que empresas grandes fazem.

Opção 3b — CouchDB multi-master

NOSQL · DOCUMENTOS

O Apache CouchDB foi projetado para replicação multi-master desde o primeiro dia. Cada nó aceita escritas; conflitos são detectados e resolvidos automaticamente via MVCC. É a opção mais elegante conceitualmente — a replicação não é um add-on, é a feature principal.

Passo a passo

# docker-compose.yml em ambos os sites
services:
  couchdb:
    image: couchdb:3.3
    restart: unless-stopped
    ports:
      - "5984:5984"
    environment:
      COUCHDB_USER: admin
      COUCHDB_PASSWORD_FILE: /run/secrets/couch_password
      NODENAME: casa-01.local   # muda em cada site
    volumes:
      - ./couchdata:/opt/couchdb/data
    secrets: [couch_password]
# Configurar replicacao continua bidirecional
# De casa-01 -> casa-02
curl -X POST http://admin:xxx@localhost:5984/_replicator \
  -H "Content-Type: application/json" \
  -d '{
    "_id": "casa01-to-casa02",
    "source": "http://admin:xxx@localhost:5984/appdb",
    "target": "http://admin:xxx@casa-02.tailnet:5984/appdb",
    "continuous": true
  }'

# E o caminho inverso, configurado em casa-02

Nuvem opcional como hub

Em vez de replicação direta site-a-site, você pode usar IBM Cloudant ou um CouchDB hospedado como hub central — cada site replica contra o hub, o que simplifica firewall e garante um backup sempre atualizado.

Resolução de conflitos

Quando dois sites escrevem no mesmo documento durante uma partição, o CouchDB escolhe uma "winning revision" deterministicamente mas preserva as outras — sua aplicação pode detectar e resolver: 

// Detectar conflitos ao ler
const doc = await db.get('doc-id', { conflicts: true });
if (doc._conflicts) {
  // Aplicar logica de merge da sua aplicacao
  // ex: ultimo timestamp vence, ou concatenar, etc
}
Vantagens
  • Multi-master de verdade — ambos os sites podem escrever simultaneamente.
  • Replicação é feature de primeira classe, não um add-on frágil.
  • Funciona offline por dias e converge quando reconecta.
  • HTTP API nativa — fácil de inspecionar e debugar.
  • PouchDB permite até mesmo replicação para o browser do cliente.
Desvantagens
  • Migrar de SQL para documentos é uma mudança grande de paradigma.
  • Queries ad-hoc são mais limitadas (Mango queries, Views pré-definidas).
  • Sem JOINs — você precisa denormalizar dados.
  • Resolução de conflitos é responsabilidade da aplicação.
  • Comunidade menor que Postgres/MySQL.
Escolha se… você está projetando uma aplicação do zero ou ela já é orientada a documentos, e valoriza muito a simplicidade operacional da replicação (ela "just works"). É especialmente brilhante se houver um app mobile/web offline-first usando PouchDB.

Opção 3c — Litestream + SQLite

SQLITE · BACKUP CONTÍNUO

O Litestream transforma um banco SQLite local em um banco com replicação contínua para S3 (ou Cloudflare R2, Backblaze B2). Importante: não é multi-master — é uma estratégia de backup-em-tempo-real com restore rápido. Funciona bem para apps onde só um site escreve por vez.

Passo a passo

# Instalar litestream
curl -LO https://github.com/benbjohnson/litestream/releases/latest/download/litestream-linux-amd64.deb
sudo dpkg -i litestream-linux-amd64.deb
# /etc/litestream.yml
dbs:
  - path: /opt/meu-app/data/appdb.sqlite
    replicas:
      - type: s3
        bucket: meu-app-db-backup
        path: appdb
        endpoint: https://<account>.r2.cloudflarestorage.com
        access-key-id: ${R2_ACCESS_KEY}
        secret-access-key: ${R2_SECRET_KEY}
        sync-interval: 1s
        retention: 72h
sudo systemctl enable --now litestream

# No outro site, antes de subir a app, restaurar o backup:
litestream restore -o /opt/meu-app/data/appdb.sqlite \
  s3://meu-app-db-backup/appdb

Fluxo de failover

  1. App em casa-01 escreve no SQLite local. Litestream replica WAL para R2 a cada 1s.
  2. Casa-01 cai. Cloudflare LB direciona tráfego para casa-02.
  3. Casa-02 detecta (via script) que agora é primária, faz litestream restore e sobe a app.
  4. Casa-01 volta — apaga seu banco local, restaura do R2 e volta como standby.
Atenção crítica Nunca deixe as duas apps escrevendo no mesmo banco SQLite ao mesmo tempo via Litestream — você perderá dados. Essa opção assume que o Cloudflare LB garante que apenas um site está ativo a qualquer momento. Implemente um lock distribuído se precisar de garantia forte.
Vantagens
  • Simplicidade extrema — é só um binário que observa o WAL.
  • SQLite é rápido, confiável e sem processo de servidor.
  • Custo baixíssimo (R2 cobra ~$0.015/GB/mês sem egress).
  • Restore em segundos, não minutos.
  • Excelente para apps pequenas/médias com poucos writes.
Desvantagens
  • Não é multi-master — só um site escreve por vez.
  • Durante failover há uma janela de ~segundos com dados stale.
  • SQLite tem limites para writes concorrentes (problema para apps heavy-write).
  • Depende de um bucket S3/R2 acessível.
  • Restore é "all-or-nothing" — não há replicação contínua no standby.
Escolha se… sua aplicação é relativamente pequena, usa SQLite (ou pode usar) e você prioriza simplicidade operacional máxima acima de tudo. É a opção com menos partes móveis — perfeita para um lab pessoal, aplicações internas ou projetos indie.

Opção 3d — rqlite ou Turso

SQLITE DISTRIBUÍDO · RAFT

O rqlite é um banco distribuído construído sobre SQLite, usando o algoritmo Raft para consenso. Você roda 3 ou mais nós (um em cada residência + um na nuvem), e ele automaticamente elege um líder, replica escritas e faz failover em segundos. O Turso é a versão comercial/SaaS baseada em libSQL (fork do SQLite) com replicação geográfica.

rqlite: cluster de 3 nós

# No no 1 (casa-01) - inicia o cluster
rqlited -node-id 1 \
  -http-addr 0.0.0.0:4001 \
  -raft-addr 0.0.0.0:4002 \
  /var/lib/rqlite

# No 2 (casa-02) - junta ao cluster
rqlited -node-id 2 \
  -http-addr 0.0.0.0:4001 \
  -raft-addr 0.0.0.0:4002 \
  -join http://casa-01.tailnet:4001 \
  /var/lib/rqlite

# No 3 (VPS em nuvem, desempata votacao) - tambem junta
rqlited -node-id 3 \
  -http-addr 0.0.0.0:4001 \
  -raft-addr 0.0.0.0:4002 \
  -join http://casa-01.tailnet:4001 \
  /var/lib/rqlite
# A app sempre fala com qualquer no - ele redireciona ao leader automaticamente
curl -XPOST "localhost:4001/db/execute?pretty" -H "Content-Type: application/json" -d '[
  ["INSERT INTO users(name) VALUES(?)", "Franklin"]
]'

Turso (alternativa gerenciada)

Se você não quer operar o cluster, o Turso oferece o mesmo modelo como serviço, com réplicas próximas geograficamente: 

turso db create meu-app
turso db replicate meu-app gru  # replica em Sao Paulo
turso db replicate meu-app scl  # replica em Santiago
turso db shell meu-app

Por que 3 nós, não 2

O Raft exige quorum (maioria) para aceitar escritas. Com apenas 2 nós, se qualquer um cair, o outro fica sem quorum e para de aceitar writes. Por isso o terceiro nó na nuvem é essencial — ele serve como "juiz de desempate" sem precisar de muitos recursos (um droplet de US$ 4/mês basta).

Vantagens
  • Failover automático em segundos, sem intervenção humana.
  • Consistência forte — garantia de linearizabilidade.
  • Não há "split-brain" graças ao Raft.
  • SQL completo (é SQLite por baixo).
  • Turso remove toda a dor operacional com US$ 0-20/mês.
Desvantagens
  • Exige pelo menos 3 nós para ser útil (custa um VPS extra).
  • Latência de writes = round-trip até o leader + quorum.
  • Se a internet da sua casa cair, você perde quorum do lado local.
  • Ecossistema ainda menor que Postgres/MySQL.
  • Turso tem vendor lock-in (embora libSQL seja open-source).
Escolha se… você quer failover genuinamente automático e consistência forte sem construir isso manualmente. Turso é a opção mais "moderna" — em 10 minutos você tem um banco replicado geograficamente. rqlite é a versão self-hosted para quem quer controle total.

08 — ValidaçãoTestes de failover

Uma arquitetura de alta disponibilidade que nunca foi testada não é confiável — é teatro. Faça pelo menos esses três cenários de teste antes de confiar em produção:

Teste 1 — Queda abrupta de energia no site primário

# Simular: desligar fisicamente o servidor da casa-01 (puxar tomada)
# Ou, menos destrutivo, desligar o docker e o cloudflared:
sudo systemctl stop docker cloudflared

# Em outra rede, monitorar o tempo ate recuperacao:
while true; do
  curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" \
    https://app.seudominio.com.br/health
  sleep 5
done

Métrica esperada: recuperação em menos de 90 segundos (health check com intervalo de 30s × 2 retries + margem).

Teste 2 — Partição de rede

Bloqueie o tráfego entre as duas residências (mas mantenha internet pública em ambas) e veja como o sistema se comporta. Este é o teste mais revelador: 

# No firewall da casa-01, bloquear o IP Tailscale da casa-02
sudo iptables -A OUTPUT -d 100.x.y.z -j DROP

# Fazer escritas nos dois sites durante a particao
# Remover o bloqueio
sudo iptables -D OUTPUT -d 100.x.y.z -j DROP

# Verificar se a replicacao converge e se ha conflitos

Teste 3 — Falha do banco mas app saudável

Derrube apenas o container do banco no site primário e veja se o /health endpoint detecta e força o failover — esse é o cenário mais sutil e o que revela endpoints de health check mal feitos.

09 — ObservabilidadeMonitoramento e alertas

Para saber que um failover aconteceu, você precisa de monitoramento externo (fora das duas residências). Setup mínimo recomendado:

  • Healthchecks.io ou UptimeRobot pingando o domínio público a cada minuto — alerta por e-mail/Telegram se o serviço cair.
  • Cloudflare Analytics — notificações quando um pool do Load Balancer fica unhealthy.
  • Prometheus + Grafana rodando numa VPS separada, coletando métricas dos dois sites via Tailscale (CPU, disco, lag de replicação, tamanho dos WAL files no Postgres).
  • Bot do Telegram customizado que avisa "🔴 casa-01 caiu, tráfego agora em casa-02" e "🟢 casa-01 voltou e está sincronizando".

10 — EntregaChecklist operacional

Antes de considerar a arquitetura "em produção", confirme cada item:

Infraestrutura

  • ☐ Duas residências com ISPs diferentes confirmadas
  • ☐ UPS dimensionado em ambos os sites (teste real de autonomia)
  • ☐ SSD NVMe com SMART OK em ambos os servidores
  • ☐ Backup off-site semanal (além da replicação) em R2/B2

Roteamento

  • ☐ Dois tunnels Cloudflare funcionando independentemente
  • ☐ Load Balancer em modo Failover configurado
  • ☐ Health check apontando para /health real (valida banco também)
  • ☐ Teste de failover documentado com tempo de recuperação medido

Banco de dados

  • ☐ Opção escolhida (3a/3b/3c/3d) implementada
  • ☐ Replicação bidirecional testada (ou direção de failover, se single-writer)
  • ☐ Procedimento de promoção de standby documentado passo-a-passo
  • ☐ Teste de partição de rede executado pelo menos uma vez
  • ☐ Restore de backup testado (não apenas criado)

Aplicação

  • ☐ Deploy via Git webhook ou cron nos dois sites
  • ☐ Variáveis de ambiente separadas por site (SITE_ID)
  • ☐ Secrets gerenciados fora do Git (docker secrets ou vault)
  • ☐ Uploads usando nomes UUID para evitar conflitos no Syncthing

Monitoramento

  • ☐ Alerta externo (Healthchecks/UptimeRobot) configurado
  • ☐ Notificações Cloudflare ativas
  • ☐ Dashboard Grafana com lag de replicação
  • ☐ Runbook escrito para "o que fazer quando X acontecer"

A confiabilidade não é um estado — é um hábito de testar as hipóteses regularmente.
