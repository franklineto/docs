Capítulo 1 — O que é o Databricks?

O Databricks é uma plataforma unificada de dados e inteligência artificial, criada em 2013 pelos criadores originais do Apache Spark — o projeto de processamento de dados que revolucionou a indústria. A empresa nasceu na Universidade de Berkeley (daí o nome Databricks) com o objetivo de tornar o Spark acessível para empresas de qualquer tamanho.

Pense no Databricks como um ambiente completo onde engenheiros de dados, cientistas de dados e analistas podem trabalhar juntos na mesma plataforma, usando as mesmas ferramentas, acessando os mesmos dados, sem precisar copiar arquivos entre sistemas ou configurar infraestrutura complexa.

1.1 O problema que o Databricks resolve

Antes do Databricks, as empresas que queriam processar grandes volumes de dados enfrentavam um conjunto frustrante de desafios:

  • Dados espalhados em dezenas de sistemas diferentes que não se comunicavam
  • Equipes de engenharia, ciência de dados e BI usando ferramentas incompatíveis
  • Infraestrutura cara e difícil de escalar — comprar servidores levava meses
  • Código de transformação de dados frágil, sem versionamento, sem testes
  • Dados duplicados em múltiplos lugares, gerando inconsistências e alto custo

O Databricks resolveu isso criando uma plataforma que roda na nuvem (AWS, Azure ou GCP), que escala automaticamente conforme a demanda e que unifica todos esses casos de uso em um único lugar.

1.2 Lakehouse: a arquitetura central

O conceito mais importante para entender o Databricks é o Lakehouse. Para chegar nele, precisamos entender dois conceitos anteriores:

Data Warehouse vs Data Lake vs Lakehouse
Data Warehouse: banco de dados altamente otimizado para análises. Dados estruturados,
consultas rápidas, mas caro e rígido. Exemplos: Snowflake, BigQuery, Redshift.
Data Lake: armazenamento barato de qualquer tipo de dado (arquivos, imagens, logs).
Flexível, mas sem organização — vira facilmente um 'data swamp' (pântano de dados).
Exemplos: S3, ADLS, GCS.
Lakehouse: combina o melhor dos dois. Armazenamento barato como o Data Lake,
com confiabilidade e performance de consulta como o Data Warehouse.
A Databricks popularizou esse conceito com o Delta Lake.

1.3 Onde o Databricks roda

O Databricks é um serviço gerenciado que roda sobre as três grandes nuvens públicas:

NuvemServiço Databricks
Amazon Web Services (AWS)Databricks on AWS
Microsoft AzureAzure Databricks
Google Cloud Platform (GCP)Databricks on GCP

Em todos os casos, o Databricks gerencia a infraestrutura por você. Você não precisa instalar nada, configurar servidores ou gerenciar atualizações. Basta criar uma conta e começar a trabalhar.

1.4 Os principais casos de uso

  • Engenharia de dados: pipelines ETL/ELT para ingerir, transformar e carregar dados
  • Machine Learning: treinar, avaliar e colocar modelos em produção
  • SQL Analytics: consultas SQL em grandes volumes de dados
  • Streaming: processamento de dados em tempo real (Kafka, Event Hubs)
  • Governança de dados: controle de acesso, linhagem e qualidade de dados

Capítulo 2 — Conceitos Fundamentais

Antes de começar a usar o Databricks, é essencial entender um conjunto de conceitos que aparecem em toda a documentação e em qualquer conversa sobre a plataforma. Este capítulo apresenta cada um deles de forma clara, com analogias do mundo real.

2.1 Apache Spark — o motor de processamento

O Apache Spark é o motor de processamento distribuído que alimenta o Databricks. Ele foi criado para superar as limitações do Hadoop MapReduce, que era lento porque escrevia resultados intermediários no disco a cada etapa do processamento.

O Spark mantém os dados na memória RAM sempre que possível, o que o torna até 100 vezes mais rápido que o Hadoop para certos tipos de operação. Imagine a diferença entre ler um livro que está na sua mesa (memória RAM) versus ter que ir até a biblioteca cada vez que precisa consultar algo (disco).

Como o Spark distribui o trabalho

O Spark usa um modelo de cluster: um conjunto de máquinas (nós) trabalhando juntas. Há dois tipos de nó:

  • Driver: o cérebro do cluster. Recebe o código, cria o plano de execução e coordena os workers.
  • Workers (Executors): os braços do cluster. Recebem partes dos dados e executam as tarefas em paralelo.

Quando você escreve df.count() em um notebook Databricks, o driver divide essa tarefa entre todos os workers disponíveis, cada um conta sua parte dos dados, e o driver soma os resultados parciais. Isso é processamento distribuído.

2.2 Workspace

O Workspace é o ambiente de trabalho do Databricks — pense nele como o seu escritório virtual. Tudo que você cria fica dentro de um Workspace: notebooks, clusters, jobs, pipelines e pastas organizacionais.

Uma empresa geralmente tem pelo menos dois Workspaces: um para desenvolvimento (onde engenheiros experimentam) e outro para produção (onde processos críticos rodam). Isso evita que um experimento mal sucedido afete dados de produção.

2.3 Cluster

Um cluster é um conjunto de máquinas virtuais que o Databricks cria na nuvem para executar seu código. Você define o tamanho (tipo e quantidade de máquinas) e o Databricks faz todo o resto: instala o Spark, configura a comunicação entre as máquinas e disponibiliza o cluster para uso.

Tipo de clusterQuando usar
All-PurposeDesenvolvimento interativo em notebooks. Pode ser iniciado e parado manualmente.
Job ClusterExecução automatizada de jobs. Criado automaticamente, destruído ao final do job.
SQL WarehouseConsultas SQL via Databricks SQL. Otimizado para queries analíticas.
Custo e auto-termination
Clusters cobram por hora de uso, mesmo que ninguém esteja rodando código.
Configure sempre o 'auto-termination' (encerramento automático) para evitar cobranças
desnecessárias. O padrão recomendado é 30 minutos de inatividade.
Clusters do tipo Job são automaticamente destruídos ao final da execução — mais econômico.

2.4 Notebook

Um notebook é um documento interativo que mistura código, resultados e texto explicativo. No Databricks, você pode escrever Python, SQL, Scala ou R no mesmo notebook — cada célula pode usar uma linguagem diferente.

Notebooks são perfeitos para exploração e análise de dados porque você vê o resultado imediatamente abaixo de cada célula, sem precisar rodar o programa inteiro para saber se uma parte funcionou.

# Exemplo de célula Python em um notebook Databricks
df = spark.read.format(&#x27;parquet').load('/mnt/dados/vendas/')
df.show(5)    # exibe as 5 primeiras linhas imediatamente
-- Exemplo de célula SQL no mesmo notebook
%sql
SELECT regiao, SUM(valor) as total
FROM vendas
WHERE ano = 2024
GROUP BY regiao
ORDER BY total DESC

2.5 Delta Lake

Delta Lake é o formato de armazenamento open-source criado pela Databricks que resolve o problema central do Data Lake: dados desorganizados, sem confiabilidade e sem suporte a operações como update e delete.

Pense no Delta Lake como um arquivo Excel, mas para bilhões de linhas. Ele mantém um histórico de todas as alterações (chamado transaction log), permite desfazer operações erradas, e garante que múltiplos processos possam ler e escrever ao mesmo tempo sem corromper os dados.

  • ACID transactions: garante que operações sejam completas ou não existam (sem dados pela metade)
  • Time travel: consultar versões anteriores dos dados com SELECT * FROM tabela VERSION AS OF 5
  • Schema enforcement: rejeita dados com colunas erradas ou tipos incompatíveis
  • Schema evolution: permite adicionar colunas novas sem quebrar pipelines existentes
  • Operações DML: suporte a UPDATE, DELETE e MERGE em arquivos Parquet

2.6 DataFrame

Um DataFrame é a estrutura de dados central no Spark e no Databricks. É uma tabela distribuída — como uma planilha Excel que está dividida em pedaços espalhados por várias máquinas, mas que parece uma única tabela quando você a consulta.

# Criando um DataFrame a partir de um arquivo CSV
df = spark.read.format(&#x27;csv') \
.option(&#x27;header', 'true') \
.option(&#x27;inferSchema', 'true') \
.load(&#x27;/mnt/dados/clientes.csv')
# Inspecionando o schema (estrutura de colunas e tipos)
df.printSchema()
# Resultado esperado:
# root
#  |-- id: integer (nullable = true)
#  |-- nome: string (nullable = true)
#  |-- valor_compra: double (nullable = true)

2.7 RDD — Resilient Distributed Dataset

O RDD é a estrutura de dados de mais baixo nível do Spark — anterior ao DataFrame. Um RDD é uma coleção distribuída e imutável de objetos. Na prática, você raramente usará RDDs diretamente no Databricks moderno, mas é importante conhecer o conceito porque toda a API de alto nível (DataFrames, SQL) é construída sobre RDDs por baixo dos panos.

A palavra Resilient (resiliente) se refere ao fato de que o Spark consegue reconstruir automaticamente partes perdidas de um RDD em caso de falha de um nó do cluster — sem perder o trabalho feito até então.

2.8 Metastore e Unity Catalog

O Metastore é o catálogo central que guarda as definições das tabelas: onde os dados ficam armazenados, quais são as colunas, seus tipos, estatísticas e permissões. É como o índice de uma biblioteca — sem ele, você saberia que tem livros mas não saberia onde estão.

O Unity Catalog é a versão moderna e unificada do Metastore no Databricks. Ele adiciona governança centralizada — um único ponto para controlar quem pode acessar o quê, em todos os Workspaces da empresa. Ele opera em três níveis hierárquicos:

  • Catalog: o nível mais alto — agrupa esquemas relacionados (ex: catalog_producao)
  • Schema (Database): agrupa tabelas relacionadas (ex: vendas, rh, financeiro)
  • Table/View: as tabelas e visões propriamente ditas
-- Acessando uma tabela no Unity Catalog
SELECT * FROM catalog_producao.vendas.pedidos LIMIT 10;
-- Criando uma tabela no Unity Catalog
CREATE TABLE catalog_producao.vendas.clientes (
id         BIGINT,
nome       STRING,
email      STRING,
criado_em  TIMESTAMP
) USING DELTA;

3.3 Databricks Runtime (DBR)

O Databricks Runtime é a versão do ambiente de execução instalada no cluster. Cada versão inclui versões específicas do Spark, Python, bibliotecas de ML e outras dependências. A nomenclatura segue o padrão:

DBR 14.3 LTS    → versão 14.3, suporte de longo prazo (mais estável)
DBR 15.2        → versão 15.2 (mais recente, pode ter breaking changes)
DBR 14.3 LTS ML → inclui bibliotecas de Machine Learning (TensorFlow, PyTorch, etc.)
DBR 14.3 LTS GPU → otimizado para GPUs (treino de modelos de deep learning)

Para aprendizado, use sempre a versão LTS mais recente. Para produção, nunca mude a versão de DBR sem testar antes — atualizações podem quebrar código existente.

4.1 SparkSession — o ponto de entrada

A SparkSession é o objeto que representa sua conexão com o Spark. No Databricks, ela já está disponível automaticamente através da variável spark — você não precisa criá-la. Em um projeto local (fora do Databricks), você precisaria instanciá-la manualmente.

4.2 Lendo dados de diversas fontes

O Spark suporta dezenas de fontes de dados. A API é sempre a mesma: spark.read.format().options().load(). Veja os exemplos mais comuns:

Lendo CSV

4.3 Transformações e Ações — a diferença crucial

Este é um dos conceitos mais importantes para entender a performance do Spark. Existem dois tipos de operações: transformações (lazy) e ações (eager).

Lazy Evaluation — execução preguiçosa
Quando você chama df.filter(...).groupBy(...).agg(...), o Spark NÃO executa nada.
Ele apenas constrói um plano de execução (DAG - Directed Acyclic Graph).
A execução real só acontece quando você chama uma AÇÃO: .show(), .count(),
.collect(), .write.save(). Isso permite ao Spark otimizar o plano inteiro antes
de processar qualquer dado — juntando operações, eliminando etapas desnecessárias.
Transformações (lazy)Ações (eager — disparam execução)
filter() / where()show() — exibe N linhas
select() / selectExpr()count() — conta linhas
groupBy() / agg()collect() — traz todos os dados para o driver
join()first() / head() — primeira linha
withColumn() / drop()write.save() — grava em disco
orderBy() / sort()toPandas() — converte para Pandas DataFrame
limit() / distinct()foreach() — aplica função a cada linha

4.4 Operações essenciais com DataFrames

Selecionando e renomeando colunas

from pyspark.sql.functions import col, expr
# select básico
df.select(&#x27;nome', 'valor', 'data')
# select com renomeação
df.select(
col(&#x27;nome').alias('nome_cliente'),
col(&#x27;valor').alias('valor_venda'),
col(&#x27;data')
)
# selectExpr — SQL dentro do Python
df.selectExpr(
&#x27;nome as nome_cliente',
&#x27;valor * 1.10 as valor_com_imposto',
&#x27;year(data) as ano'
)

Filtrando dados

# filter e where são equivalentes
df.filter(col(&#x27;valor') > 1000)
df.where(&#x27;valor > 1000')   # SQL string também funciona
# múltiplos filtros
df.filter(
(col(&#x27;valor') > 1000) &
(col(&#x27;status') == 'aprovado') &
(col(&#x27;data') >= '2024-01-01')
)
# filtro com isin (equivalente ao IN do SQL)
df.filter(col(&#x27;regiao').isin(['Sul', 'Sudeste', 'Centro-Oeste']))
# filtro por nulos
df.filter(col(&#x27;email').isNull())     # linhas sem email
df.filter(col(&#x27;email').isNotNull())  # linhas com email

Adicionando e modificando colunas

from pyspark.sql.functions import lit, when, upper, to_date, datediff, current_date
# adicionando coluna constante
df.withColumn(&#x27;fonte', lit('sistema_vendas'))
# adicionando coluna calculada
df.withColumn(&#x27;valor_com_taxa', col('valor') * 1.05)
# coluna condicional (CASE WHEN do SQL)
df.withColumn(&#x27;categoria_valor',
when(col(&#x27;valor') < 100, 'baixo')
.when(col(&#x27;valor') < 1000, 'medio')
.otherwise(&#x27;alto')
)
# convertendo tipos
df.withColumn(&#x27;data', to_date(col('data_str'), 'dd/MM/yyyy'))
# calculando diferença de datas
df.withColumn(&#x27;dias_desde_compra',
datediff(current_date(), col(&#x27;data_compra'))
)

Agregações

from pyspark.sql.functions import sum, avg, count, min, max, countDistinct
# groupBy simples
df.groupBy(&#x27;regiao') \
.agg(
sum(&#x27;valor').alias('total_vendas'),
avg(&#x27;valor').alias('ticket_medio'),
count(&#x27;*').alias('qtd_pedidos'),
countDistinct(&#x27;cliente_id').alias('clientes_unicos')
)
# múltiplos groupBy
df.groupBy(&#x27;regiao', 'ano', 'mes') \
.agg(sum(&#x27;valor').alias('total'))
# pivot — transforma valores de uma coluna em colunas
df.groupBy(&#x27;regiao') \
.pivot(&#x27;ano', [2022, 2023, 2024]) \
.agg(sum(&#x27;valor'))

4.5 Joins entre DataFrames

Joins no Spark funcionam como joins no SQL, mas com algumas nuances importantes de performance que veremos no capítulo de otimização.

# inner join (padrão)
df_pedidos.join(df_clientes, on=&#x27;cliente_id', how='inner')
# quando as colunas têm nomes diferentes
df_pedidos.join(df_clientes,
df_pedidos.cli_id == df_clientes.id,
how=&#x27;left'
)
# tipos de join: inner, left, right, full, left_semi, left_anti, cross
# left_semi: retorna apenas linhas do lado esquerdo que têm correspondência
# equivale a: SELECT * FROM pedidos WHERE cliente_id IN (SELECT id FROM clientes)
df_pedidos.join(df_clientes, on=&#x27;cliente_id', how='left_semi')
# left_anti: retorna linhas do lado esquerdo SEM correspondência
# equivale a: SELECT * FROM pedidos WHERE cliente_id NOT IN (SELECT id FROM clientes)
df_pedidos.join(df_clientes, on=&#x27;cliente_id', how='left_anti')

Capítulo 5 — Delta Lake em Profundidade

O Delta Lake é o coração da arquitetura Lakehouse no Databricks. Entendê-lo bem é fundamental para criar pipelines confiáveis e eficientes. Neste capítulo, vamos explorar cada funcionalidade com exemplos práticos.

5.1 Criando tabelas Delta

Existem três formas principais de criar uma tabela Delta no Databricks:

1. A partir de um DataFrame

# Salva o DataFrame como tabela Delta
df.write \
.format(&#x27;delta') \
.mode(&#x27;overwrite')   \  # ou 'append', 'error', 'ignore'
.option(&#x27;overwriteSchema', 'true') \
.saveAsTable(&#x27;vendas.pedidos')
# Salvando em um path específico (sem registrar no metastore)
df.write.format(&#x27;delta').mode('overwrite').save('/mnt/gold/pedidos/')

2. Via SQL CREATE TABLE

%sql
CREATE TABLE IF NOT EXISTS vendas.pedidos (
id           BIGINT     NOT NULL,
cliente_id   BIGINT     NOT NULL,
valor        DECIMAL(12,2),
status       STRING,
criado_em    TIMESTAMP  DEFAULT current_timestamp()
)
USING DELTA
PARTITIONED BY (ano INT, mes INT)
LOCATION &#x27;/mnt/gold/pedidos/'
COMMENT &#x27;Tabela de pedidos de venda';

5.2 Operações DML — Update, Delete, Merge

Uma das maiores vantagens do Delta Lake sobre Parquet puro é o suporte nativo a operações de atualização e exclusão, que em um Data Lake tradicional são extremamente difíceis de implementar.

UPDATE

%sql
-- Atualiza o status de pedidos com mais de 30 dias sem entrega
UPDATE vendas.pedidos
SET    status = &#x27;cancelado_atraso'
WHERE  status = &#x27;enviado'
AND    datediff(current_date(), criado_em) > 30;

DELETE

%sql
-- Remove pedidos de teste
DELETE FROM vendas.pedidos
WHERE cliente_id IN (SELECT id FROM clientes.testes);

MERGE (upsert) — a operação mais poderosa

O MERGE atualiza registros existentes e insere novos em uma única operação atômica. É o padrão ouro para pipelines de dados incrementais — você não precisa apagar e recarregar a tabela inteira quando quer atualizar alguns registros.

%sql
MERGE INTO vendas.clientes AS destino
USING (SELECT * FROM staging.clientes_novos) AS origem
ON destino.id = origem.id
-- se encontrou correspondência: atualiza
WHEN MATCHED AND origem.email != destino.email THEN
UPDATE SET
destino.email        = origem.email,
destino.atualizado_em = current_timestamp()
-- se não encontrou: insere como novo
WHEN NOT MATCHED THEN
INSERT (id, nome, email, criado_em)
VALUES (origem.id, origem.nome, origem.email, current_timestamp())
-- se está no destino mas não na origem: pode deletar (opcional)
WHEN NOT MATCHED BY SOURCE AND destino.ativo = true THEN
UPDATE SET destino.ativo = false;

5.3 Time Travel — viagem no tempo

O Delta Lake mantém um histórico completo de todas as alterações da tabela. Isso permite consultar versões anteriores dos dados, o que é extremamente útil para auditorias, depuração de erros e recuperação de dados deletados acidentalmente.

%sql
-- Ver o histórico da tabela
DESCRIBE HISTORY vendas.pedidos;
-- Consultar como a tabela estava ontem
SELECT * FROM vendas.pedidos TIMESTAMP AS OF &#x27;2024-06-15 09:00:00';
-- Consultar versão específica (número da versão do DESCRIBE HISTORY)
SELECT * FROM vendas.pedidos VERSION AS OF 42;
-- Restaurar para uma versão anterior (CUIDADO: irreversível!)
RESTORE TABLE vendas.pedidos TO VERSION AS OF 42;
-- Python equivalente
df_ontem = spark.read \
.format(&#x27;delta') \
.option(&#x27;timestampAsOf', '2024-06-15') \
.table(&#x27;vendas.pedidos')

5.4 OPTIMIZE e VACUUM

Com o uso intenso, tabelas Delta acumulam muitos arquivos pequenos, o que degrada a performance de leitura. O OPTIMIZE resolve isso juntando arquivos pequenos em arquivos maiores.

%sql
-- Compacta arquivos pequenos da tabela (recomendado: rodar semanalmente)
OPTIMIZE vendas.pedidos;
-- OPTIMIZE com Z-ORDER: ordena fisicamente os dados por coluna(s)
-- Acelera muito filtros e joins nessas colunas
OPTIMIZE vendas.pedidos ZORDER BY (cliente_id, criado_em);

O VACUUM remove arquivos físicos antigos que não são mais necessários. Por padrão, o Delta mantém 7 dias de histórico. O VACUUM deleta arquivos com mais de N horas de histórico — isso afeta o time travel.

%sql
-- Remove arquivos com mais de 7 dias (padrão de retenção)
VACUUM vendas.pedidos;
-- Remove arquivos com mais de 168 horas (7 dias)
VACUUM vendas.pedidos RETAIN 168 HOURS;
-- NUNCA faça VACUUM com menos de 7 dias em tabelas com leitores concorrentes
-- Pode causar erros de &#x27;file not found' em jobs em execução

5.5 Constraints e Schema Enforcement

O Delta Lake permite definir constraints (restrições) nas tabelas para garantir a qualidade dos dados. Dados que violem as constraints são rejeitados com um erro, impedindo que dados inválidos entrem na tabela.

%sql
-- Adiciona constraint de NOT NULL
ALTER TABLE vendas.pedidos
ADD CONSTRAINT pedidos_valor_positivo CHECK (valor > 0);
-- Adiciona constraint de domínio
ALTER TABLE vendas.pedidos
ADD CONSTRAINT pedidos_status_valido
CHECK (status IN (&#x27;novo', 'aprovado', 'enviado', 'entregue', 'cancelado'));
-- Lista constraints da tabela
DESCRIBE DETAIL vendas.pedidos;
-- Remove constraint
ALTER TABLE vendas.pedidos DROP CONSTRAINT pedidos_valor_positivo;

6.2 SQL analítico avançado

O Databricks SQL suporta toda a sintaxe ANSI SQL mais extensões poderosas. Aqui estão os padrões mais importantes:

Window Functions (Funções de Janela)

Window functions calculam valores com base em um conjunto de linhas relacionadas à linha atual, sem colapsar o resultado em uma única linha (como o GROUP BY faz). São essenciais para análises sequenciais e de ranking.

%sql
SELECT
cliente_id,
data_pedido,
valor,
-- Número sequencial do pedido por cliente
ROW_NUMBER() OVER (
PARTITION BY cliente_id ORDER BY data_pedido
) AS numero_pedido,
-- Valor acumulado de compras por cliente
SUM(valor) OVER (
PARTITION BY cliente_id
ORDER BY data_pedido
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) AS total_acumulado,
-- Diferença para o pedido anterior do mesmo cliente
valor - LAG(valor, 1, 0) OVER (
PARTITION BY cliente_id ORDER BY data_pedido
) AS variacao_vs_anterior,
-- Rank por valor dentro de cada mês
DENSE_RANK() OVER (
PARTITION BY year(data_pedido), month(data_pedido)
ORDER BY valor DESC
) AS rank_valor_mes
FROM vendas.pedidos
WHERE status = &#x27;entregue';

CTEs — Common Table Expressions

CTEs (WITH ...) tornam queries complexas legíveis ao dar nomes a subqueries intermediárias. São especialmente úteis quando a mesma subquery é referenciada múltiplas vezes.

%sql
WITH
-- CTE 1: resumo de vendas por cliente
vendas_cliente AS (
SELECT
cliente_id,
COUNT(*)   AS total_pedidos,
SUM(valor) AS total_gasto,
MAX(data_pedido) AS ultima_compra
FROM vendas.pedidos
WHERE status = &#x27;entregue'
GROUP BY cliente_id
),
-- CTE 2: classificação RFM (Recência, Frequência, Monetário)
rfm AS (
SELECT
cliente_id,
total_pedidos,
total_gasto,
datediff(current_date(), ultima_compra) AS dias_sem_compra,
CASE
WHEN total_gasto > 5000 THEN &#x27;A'
WHEN total_gasto > 1000 THEN &#x27;B'
ELSE &#x27;C'
END AS segmento
FROM vendas_cliente
)
SELECT r.*, c.nome, c.email
FROM rfm r
JOIN clientes.base c ON r.cliente_id = c.id
WHERE r.segmento = &#x27;A'
ORDER BY r.total_gasto DESC;

6.3 Criando e gerenciando Views

Views são consultas SQL salvas que se comportam como tabelas. São úteis para simplificar queries complexas, aplicar regras de segurança (mostrando apenas certas colunas para certos usuários) e criar camadas de abstração sobre os dados brutos.

%sql
-- View simples
CREATE OR REPLACE VIEW vendas.v_pedidos_ativos AS
SELECT id, cliente_id, valor, status, criado_em
FROM vendas.pedidos
WHERE status NOT IN (&#x27;cancelado', 'rascunho');
-- View materializada (apenas em Pro/Enterprise)
-- Mantém o resultado em cache e atualiza automaticamente
CREATE MATERIALIZED VIEW vendas.v_resumo_diario AS
SELECT
date(criado_em) AS data,
COUNT(*)        AS total_pedidos,
SUM(valor)      AS receita_total
FROM vendas.pedidos
WHERE status = &#x27;entregue'
GROUP BY 1;
-- Dynamic Views — restringe dados por usuário (Unity Catalog)
CREATE VIEW vendas.v_pedidos_seguro AS
SELECT
id,
cliente_id,
CASE WHEN is_member(&#x27;grupo_financeiro')
THEN valor
ELSE NULL
END AS valor,  -- oculta valor para não-financeiros
status
FROM vendas.pedidos;

Capítulo 7 — Engenharia de Dados e Pipelines

Engenharia de dados é a disciplina de construir sistemas confiáveis para mover e transformar dados de fontes diversas até um destino onde analistas e modelos de ML possam consumi-los. O Databricks oferece três abordagens principais para isso.

7.1 Arquitetura Medalhão — Bronze, Silver, Gold

A arquitetura Medalhão é o padrão recomendado pela Databricks para organizar pipelines de dados no Lakehouse. Divide os dados em três camadas com qualidade crescente:

As três camadas da arquitetura Medalhão
BRONZE (Raw): dados brutos como chegam da fonte. Sem transformação.
Objetivo: ter uma cópia exata dos dados originais para reprocessamento.
Formato: Delta. Particionado por data de ingestão.
SILVER (Refined): dados limpos e validados. Sem duplicatas, com tipos corretos,
com registros inválidos removidos ou corrigidos. Joins simples aplicados.
GOLD (Curated): dados agregados e prontos para consumo. Tabelas de fato e
dimensão, KPIs calculados, visões de negócio. Consumidos por BI e ML.

7.2 Auto Loader — ingestão incremental inteligente

O Auto Loader é o mecanismo de ingestão incremental nativa do Databricks. Ele monitora automaticamente um diretório de armazenamento em nuvem (S3, ADLS, GCS) e processa apenas os arquivos novos desde a última execução — sem precisar listar todos os arquivos a cada run, o que seria inviável em pastas com milhões de arquivos.

Internamente, o Auto Loader usa duas estratégias dependendo do volume de arquivos: para pastas menores, usa listagem diretória com rastreamento de state; para pastas com milhões de arquivos, usa notificações de eventos do storage (SQS no AWS, Event Grid no Azure) — muito mais eficiente.

Por que usar Auto Loader em vez de spark.read simples?
spark.read.load() lê TODOS os arquivos da pasta a cada execução — sem memória do que já foi processado.
Se você tem 10 TB históricos e chegam 10 GB novos por dia, o spark.read reprocessa os 10 TB todos os dias.
O Auto Loader rastreia exatamente quais arquivos já foram lidos (via checkpoint)
e processa apenas os arquivos NOVOS desde o último run. Isso reduz custo e tempo drasticamente.
Além disso, o Auto Loader detecta e evolui o schema automaticamente quando novos campos chegam.

Configurações essenciais do Auto Loader

# Configurações do Auto Loader por cloudFiles.option()
# Formato dos arquivos de entrada
# Suporta: json, csv, parquet, avro, orc, text, binaryFile
.option(&#x27;cloudFiles.format', 'json')
# Onde guardar o schema inferido (obrigatório)
# Persiste o schema descoberto para não re-inferir a cada run
.option(&#x27;cloudFiles.schemaLocation', '/mnt/bronze/_schemas/pedidos/')
# Inferência automática de tipos (ex: '123' vira IntegerType, não StringType)
.option(&#x27;cloudFiles.inferColumnTypes', 'true')
# O que fazer quando novos campos chegam nos arquivos
# 'addNewColumns'  — adiciona a coluna nova automaticamente (recomendado)
# 'rescue'         — coloca campos inesperados em _rescued_data (JSON)
# 'failOnNewColumns' — falha o pipeline se houver campo novo (mais rígido)
.option(&#x27;cloudFiles.schemaEvolutionMode', 'addNewColumns')
# Notificações de evento (mais eficiente para pastas grandes)
.option(&#x27;cloudFiles.useNotifications', 'true')
# Filtro por extensão (evita processar arquivos temporários .tmp, .crc)
.option(&#x27;pathGlobFilter', '*.json')
# Inclui arquivos de subpastas automaticamente
.option(&#x27;recursiveFileLookup', 'true')

7.2 Delta Live Tables (DLT)

Delta Live Tables é a evolução das pipelines no Databricks. Em vez de escrever código imperativo (faça isso, depois aquilo), você declara o que cada tabela deve conter e o DLT cuida automaticamente da orquestração, da execução incremental, do tratamento de erros e da qualidade dos dados.

# Exemplo de pipeline DLT em Python
import dlt
from pyspark.sql.functions import *
# Tabela Bronze — lê da fonte
@dlt.table(
name=&#x27;bronze_pedidos',
comment=&#x27;Dados brutos de pedidos do sistema de vendas'
)
def bronze_pedidos():
return (
spark.readStream
.format(&#x27;cloudFiles')
.option(&#x27;cloudFiles.format', 'json')
.load(&#x27;/mnt/landing/pedidos/')
)
# Tabela Silver — com expectations (qualidade de dados)
@dlt.table(
name=&#x27;silver_pedidos',
comment=&#x27;Pedidos limpos e validados'
)
@dlt.expect_all_or_drop({
&#x27;id nao nulo':     'id IS NOT NULL',
&#x27;valor positivo':  'valor > 0',
&#x27;status valido':   "status IN ('novo','aprovado','enviado','entregue','cancelado')"
})
def silver_pedidos():
return (
dlt.read_stream(&#x27;bronze_pedidos')
.dropDuplicates([&#x27;id'])
.withColumn(&#x27;valor', col('valor').cast('decimal(12,2)'))
)
# Tabela Gold
@dlt.table(name=&#x27;gold_kpi_vendas')
def gold_kpi_vendas():
return (
dlt.read(&#x27;silver_pedidos')
.groupBy(year(&#x27;criado_em').alias('ano'), month('criado_em').alias('mes'))
.agg(sum(&#x27;valor').alias('receita'), count('*').alias('total'))
)
Expectations (Qualidade de Dados no DLT)
@dlt.expect('nome', 'condição') — registra violações mas não as descarta
@dlt.expect_or_drop('nome', 'condição') — descarta linhas que violam
@dlt.expect_or_fail('nome', 'condição') — falha o pipeline inteiro na violação
@dlt.expect_all_or_drop({'nome': 'condição', ...}) — múltiplas regras
As métricas de qualidade ficam disponíveis no painel de monitoramento do DLT.

7.4 Lakeflow Jobs — orquestração de pipelines

O Lakeflow Jobs (anteriormente chamado de Databricks Workflows) é a solução nativa de orquestração do Databricks. Ele permite criar pipelines de dados complexos com múltiplas tarefas interdependentes, agendamento, retry automático e monitoramento centralizado — tudo sem precisar de ferramentas externas como Airflow ou Prefect.

A grande vantagem do Lakeflow Jobs em relação a orquestradores externos é a integração profunda com o Databricks: acesso nativo ao Unity Catalog, rastreamento automático de linhagem de dados, monitoramento de qualidade via DLT, e custo otimizado com Job Clusters que sobem e morrem automaticamente.

Conceitos fundamentais do Lakeflow Jobs

ConceitoDescrição
JobContainer que agrupa uma ou mais Tasks relacionadas
TaskUnidade de trabalho: notebook, script, DLT pipeline, SQL, Python wheel
DependênciaDefine que Task B só inicia após Task A concluir (com sucesso ou erro)
ClusterJob Cluster (efêmero) ou All-Purpose Cluster (já existente)

Criando um Job via interface

  • No menu lateral, clique em 'Workflows' e depois em 'Create job'
  • Dê um nome descritivo ao Job (ex: 'pipeline_vendas_diario')
  • Configure a primeira Task: escolha o tipo (Notebook, Python, DLT, SQL, etc.)
  • Selecione o cluster — prefira Job Cluster para produção (mais econômico)
  • Adicione Tasks adicionais e defina as dependências entre elas
  • Configure o Schedule (ex: diariamente às 06:00 BRT)
  • Adicione alertas de e-mail para falhas
  • Clique em 'Create' e use 'Run now' para testar

MLflow é uma plataforma open-source para o ciclo de vida do ML, criada pela Databricks. No Databricks, ela está integrada nativamente e não requer nenhuma configuração adicional. Ela resolve o problema clássico: 'qual versão do modelo, com quais parâmetros, produziu qual resultado?'

import mlflow

# Carrega dados

Capítulo 10 — Performance e Otimização

À medida que seus dados crescem, a diferença entre um código bem escrito e um mal escrito pode ser de horas versus minutos. Este capítulo cobre as otimizações mais impactantes no Databricks.

10.1 Particionamento

Particionar uma tabela significa dividir os dados em pastas baseadas em valores de colunas específicas. Quando você filtra por uma coluna de partição, o Spark lê apenas os arquivos daquela partição — ignorando todo o resto. Isso é chamado de partition pruning.

# Escrevendo com particionamento por data
df.write \
.format(&#x27;delta') \
.partitionBy(&#x27;ano', 'mes') \
.mode(&#x27;overwrite') \
.save(&#x27;/mnt/gold/pedidos/')
# A estrutura de pastas fica:
# /mnt/gold/pedidos/ano=2024/mes=1/
# /mnt/gold/pedidos/ano=2024/mes=2/
# ...
# Esta query lê APENAS a pasta ano=2024/mes=6/ — muito mais rápido!
df = spark.read.format(&#x27;delta').load('/mnt/gold/pedidos/')
df.filter(&#x27;ano = 2024 AND mes = 6').count()
Cuidado com over-partitioning
Particionar em excesso é tão ruim quanto não particionar.
Se você particionar por 'cliente_id' e tiver 10 milhões de clientes,
vai criar 10 milhões de pastas com arquivos minúsculos — lentíssimo.
Regra de ouro: particione apenas por colunas de alta cardinalidade baixa
que são frequentemente usadas em filtros (data, status, região).
Evite particionar por IDs, timestamps precisos ou e-mails.

10.2 Broadcast Join

Quando você faz um join entre uma tabela grande e uma tabela pequena, o Spark normalmente embaralha (shuffle) os dados de ambas as tabelas pelas máquinas do cluster — uma operação cara. O Broadcast Join evita isso: a tabela pequena é copiada inteira para cada worker, eliminando o shuffle.

from pyspark.sql.functions import broadcast
# Tabela de produtos tem apenas 1.000 linhas — boa candidata para broadcast
df_pedidos_detalhes = df_pedidos.join(
broadcast(df_produtos),   # envia df_produtos inteiro para cada worker
on=&#x27;produto_id',
how=&#x27;left'
)
# O Spark também decide automaticamente fazer broadcast quando
# a tabela é menor que spark.sql.autoBroadcastJoinThreshold (padrão: 10MB)
# Para tabelas maiores, force manualmente com broadcast()

10.3 Caching

Quando você usa o mesmo DataFrame múltiplas vezes em um notebook (para exploração, diferentes análises, etc.), o Spark recalcula tudo do zero a cada vez. O cache resolve isso armazenando os dados na memória ou disco do cluster.

# Cacheia em memória (mais rápido, mas ocupa RAM)
df.cache()  # ou df.persist(StorageLevel.MEMORY_ONLY)
# Cacheia em memória + disco (mais robusto para dados maiores)
from pyspark import StorageLevel
df.persist(StorageLevel.MEMORY_AND_DISK)
# Força a materialização do cache (cache é lazy por padrão)
df.count()   # qualquer ação dispara a materialização
# Use o DataFrame quantas vezes precisar — lê do cache
df.filter(&#x27;status = "aprovado"').show()
df.groupBy(&#x27;regiao').count().show()
# Limpa o cache quando não precisar mais (libera memória!)
df.unpersist()

10.4 Evitando Shuffles desnecessários

O shuffle é a operação mais cara do Spark — é quando ele precisa redistribuir dados entre os workers da rede. Ele acontece em groupBy, join, distinct e orderBy. Algumas técnicas para minimizá-lo:

  • Filtre os dados ANTES do join — reduza o volume antes de redistribuir
  • Use broadcast join para tabelas pequenas (< 200MB)
  • Ajuste spark.sql.shuffle.partitions (padrão 200, pode ser muito para dados pequenos)
  • Use repartition() para consolidar partições antes de writes grandes
  • Use coalesce() para reduzir partições sem fazer shuffle
# Ajustando partições para o tamanho dos dados
spark.conf.set(&#x27;spark.sql.shuffle.partitions', '50')  # para dados < 1GB
# repartition: redistribui dados (faz shuffle, mas útil antes de writes)
df.repartition(10).write.format(&#x27;delta').save('/mnt/output/')
# coalesce: reduz partições SEM shuffle (só combina partições existentes)
df.coalesce(1).write.format(&#x27;delta').save('/mnt/output_single_file/')

10.5 Liquid Clustering — a nova geração do particionamento

O Liquid Clustering é a funcionalidade mais recente da Databricks para otimização de leitura (disponível no DBR 13.3+). Ele substitui o particionamento estático por um sistema que reorganiza fisicamente os dados de forma inteligente e incremental — sem os problemas do over-partitioning.

%sql
-- Criar tabela com Liquid Clustering

USING DELTA
CLUSTER BY (cliente_id, data_pedido)  -- colunas mais usadas em filtros
AS SELECT * FROM vendas.pedidos;
-- Executar o clustering (reagrupa os dados fisicamente)

-- Verificar status do clustering
DESCRIBE DETAIL vendas.pedidos_clusterizado;

Capítulo 11 — Segurança e Governança

Em ambientes corporativos, controlar quem pode ver e modificar quais dados é tão importante quanto processar os dados corretamente. O Unity Catalog do Databricks é o mecanismo central de governança.

11.1 Unity Catalog — controle de acesso

O Unity Catalog usa um modelo de permissões baseado em GRANT/REVOKE familiar para quem conhece SQL. As permissões podem ser aplicadas em qualquer nível da hierarquia (catalog, schema, table, column).

%sql
-- Concede acesso de leitura a um grupo
GRANT SELECT ON TABLE vendas.pedidos TO `grupo-analistas`;
-- Concede acesso de escrita
GRANT MODIFY ON TABLE vendas.pedidos TO `grupo-engenheiros`;
-- Concede acesso a um schema inteiro
GRANT USE SCHEMA, SELECT ON SCHEMA vendas TO `grupo-analistas`;
-- Revoga permissão
REVOKE SELECT ON TABLE vendas.clientes FROM `grupo-externo`;
-- Verifica permissões de uma tabela
SHOW GRANTS ON TABLE vendas.pedidos;

11.2 Row-Level e Column-Level Security

O Unity Catalog permite criar políticas de acesso que restringem quais LINHAS ou COLUNAS um usuário pode ver, sem precisar criar tabelas separadas para cada grupo.

%sql
-- Column Masking: oculta CPF para usuários que não são do RH
CREATE FUNCTION mascara_cpf(cpf STRING)
RETURN CASE WHEN is_member(&#x27;rh-confidencial')
THEN cpf
ELSE regexp_replace(cpf, &#x27;\\d{3}\\.\\d{3}\\.\\d{3}', '***.***.***')
END;
ALTER TABLE rh.funcionarios
ALTER COLUMN cpf SET MASK mascara_cpf;
-- Row Filter: filtra linhas baseado em atributo do usuário
CREATE FUNCTION filtro_regiao(regiao STRING)
RETURN is_member(&#x27;admin') OR regiao = current_user_attribute('regiao');
ALTER TABLE vendas.pedidos
SET ROW FILTER filtro_regiao ON (regiao);

11.4 Delta Sharing — compartilhamento seguro de dados

Delta Sharing é um protocolo open-source criado pela Databricks para compartilhar dados com segurança entre organizações diferentes — sem precisar copiar os dados, sem dar acesso ao Workspace da empresa provedora, e sem exigir que o destinatário use Databricks ou qualquer outra ferramenta específica.

Pense no Delta Sharing como uma API de dados somente leitura. A empresa provedora decide exatamente quais tabelas, schemas ou até partições específicas ela quer expor. O destinatário acessa esses dados via um link de ativação, usando qualquer ferramenta compatível com o protocolo Delta Sharing — Spark, pandas, Power BI, Tableau, entre outros.

O problema que o Delta Sharing resolve
Cenário comum: empresa A precisa fornecer dados para empresa B (parceiro, auditor, subsidiária).
Abordagem tradicional: exportar para CSV/Parquet, copiar via SFTP ou S3, empresa B reimporta.
Problemas: dados desatualizados, custo de storage duplicado, risco de vazamento na transferência,
sem controle de quem acessou o quê.
Com Delta Sharing: empresa A cria um Share no Unity Catalog, adiciona as tabelas desejadas,
cria um Recipient e gera um Activation Link. Empresa B usa o link para acessar os dados
diretamente — sempre atualizados, sem cópia, com auditoria completa de acesso.

Conceitos fundamentais do Delta Sharing

ConceitoDescrição
ShareContainer que agrupa os dados a serem compartilhados. Pode ter N tabelas de N schemas.
Schema (no Share)Agrupamento lógico de tabelas dentro do Share — não precisa ser igual ao schema original.
TableTabela Delta adicionada ao Share. Pode expor apenas certas partições (partition filter).
RecipientEntidade (empresa, sistema ou usuário) que receberá acesso ao Share.
Activation LinkURL única gerada para o Recipient ativar o acesso e baixar as credenciais.
CredentialToken de acesso gerado após ativação — usado para autenticar nas chamadas de leitura.
Open SharingCompartilhamento com qualquer ferramenta via protocolo aberto (sem precisar do Databricks).
Databricks-to-DatabricksCompartilhamento entre Workspaces Databricks — mais integrado, via Unity Catalog.

Criando um Share e adicionando tabelas (provedor)

%sql
-- Passo 1: Criar o Share
CREATE SHARE IF NOT EXISTS share_parceiro_logistica
COMMENT &#x27;Dados de pedidos compartilhados com a empresa parceira de logística';
-- Passo 2: Adicionar tabelas ao Share
-- Adiciona a tabela completa
ALTER SHARE share_parceiro_logistica
ADD TABLE vendas.pedidos;
-- Adiciona tabela com nome diferente para o destinatário
ALTER SHARE share_parceiro_logistica
ADD TABLE vendas.pedidos
AS logistica.pedidos_para_entrega;
-- Adiciona apenas partições específicas (ex: só pedidos &#x27;enviado')
ALTER SHARE share_parceiro_logistica
ADD TABLE vendas.pedidos
PARTITION (status = &#x27;enviado');
-- Adiciona um schema inteiro de uma vez

Quando o destinatário também usa Databricks, o fluxo é ainda mais integrado — não é necessário gerenciar arquivos de credenciais manualmente. O destinatário adiciona o Share diretamente ao seu Unity Catalog como um 'Foreign Catalog'.

provedor.share_parceiro_logistica;

Delta Sharing vs Replicação tradicional — resumo comparativo
Replicação: dados copiados para o ambiente do destinatário.
→ Custo dobrado de storage, dados desatualizados entre sincronizações,
risco de vazamento na transferência, sem controle pós-entrega.
Delta Sharing: dados acessados diretamente na fonte, sem cópia.
→ Sempre atualizado, custo zero de storage extra para o destinatário,
transferência criptografada, revogação instantânea de acesso,
auditoria completa de quem acessou o quê e quando.
Funciona com qualquer ferramenta: Spark, pandas, Power BI, Tableau, R.

Capítulo 12 — Processamento em Streaming

Streaming é o processamento de dados em tempo real ou quase real — à medida que chegam, em vez de esperar acumular um lote. O Databricks usa Structured Streaming, que usa a mesma API do Spark SQL para dados em tempo real e em batch.

12.1 Structured Streaming — conceitos

No Structured Streaming, um stream é como uma tabela que cresce infinitamente. A cada micro-lote (micro-batch), o Spark processa as novas linhas chegadas e atualiza o resultado. Isso é chamado de modelo de tabela incremental.

Trigger modes — quando processar?
ProcessingTime('5 seconds'): processa a cada N segundos (micro-batch padrão)
AvailableNow: processa tudo disponível e para (equivalente a batch incremental)
Once: processa um único micro-batch e para (legado, prefira AvailableNow)
Continuous('1 second'): latência muito baixa (~ms), limitado em operações

Agregações com janelas de tempo

from pyspark.sql.functions import window
# Conta pedidos por janela de 5 minutos
df_agg = (
df_pedidos_stream
.withColumn(&#x27;criado_em', to_timestamp('criado_em'))
.groupBy(

&#x27;status'

13.1 Boas práticas de desenvolvimento

Notebooks

  • Mantenha notebooks focados em uma única responsabilidade (ingestão, transformação ou análise)
  • Use comentários Markdown (células com %md) para documentar o propósito de cada seção
  • Evite hardcode de paths e parâmetros — use widgets ou parâmetros de Job
  • Versione notebooks no Git (Databricks tem integração nativa com GitHub, GitLab, Bitbucket)
# Usando widgets para parametrização
dbutils.widgets.text(&#x27;data_inicio', '2024-01-01', 'Data início:')
dbutils.widgets.dropdown(&#x27;ambiente', 'dev', ['dev', 'staging', 'prod'])

ambiente    = dbutils.widgets.get(&#x27;ambiente')
tabela = f&#x27;{ambiente}.vendas.pedidos'
df = spark.table(tabela).filter(f&#x27;data >= "{data_inicio}"')

Desempenho

  • Sempre filtre antes de fazer join — reduza o volume de dados o mais cedo possível
  • Prefira Delta Lake a Parquet puro — OPTIMIZE e Z-ORDER compensam o overhead
  • Use AQE (Adaptive Query Execution) — está habilitado por padrão no DBR 7.3+
  • Monitore o Spark UI para identificar stages lentos e skew de dados
  • Evite collect() e toPandas() em DataFrames grandes — traz tudo para o driver

Segurança

  • Nunca coloque credenciais em código — sempre use Databricks Secrets
  • Use Unity Catalog para controle granular de acesso a dados
  • Revise permissões regularmente — siga o princípio do mínimo privilégio
  • Ative auditoria (Audit Logs) para rastrear quem acessou o quê e quando
