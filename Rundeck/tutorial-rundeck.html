§ 01 — Introdução

Contexto e arquitetura

O Rundeck é uma plataforma de automação de runbooks mantida pela PagerDuty. Em uma frase: ele permite que equipes de operações transformem procedimentos ad-hoc (rodar scripts em dezenas de servidores, reiniciar serviços, aplicar correções) em jobs auditáveis, versionados e delegáveis — com controle fino de permissões e log completo de execuções.

Este guia cobre uma instalação pensada para produção, não apenas um "hello world". A diferença prática: em vez do banco H2 embutido e da senha padrão admin/admin, você terminará com PostgreSQL externo, TLS válido, LDAP e backup configurado.

Topologia alvo

Ao final deste tutorial, sua instalação terá os seguintes componentes:

  • Servidor Rundeck rodando como serviço systemd, escutando apenas em 127.0.0.1:4440.
  • Nginx como reverse proxy, encerrando TLS na porta 443 com certificado Let's Encrypt.
  • PostgreSQL (local ou remoto) armazenando jobs, execuções e metadados.
  • UFW restringindo o acesso externo apenas às portas 22, 80 e 443.
  • LDAP/AD (opcional) para autenticação corporativa.
Nota sobre edições

O Rundeck existe em duas edições: Community (gratuita, código aberto) e Enterprise (paga, com cluster, SSO avançado, alta disponibilidade). Este guia cobre a edição Community, que é suficiente para a maioria dos cenários de equipes pequenas e médias.

§ 02 — Pré-requisitos

Pré-requisitos do servidor

Antes de qualquer instalação, valide o dimensionamento mínimo. O Rundeck é uma aplicação Java/Grails — ela reserva memória de forma relativamente agressiva.

RecursoMínimoRecomendado (produção)
CPU2 vCPUs4 vCPUs
Memória4 GB RAM8 GB RAM ou mais
Disco20 GB50 GB+ (logs crescem)
SOUbuntu 22.04 LTSUbuntu 24.04 LTS
JavaOpenJDK 11OpenJDK 17

Atualize o sistema antes de começar

$ sudo apt update && sudo apt upgrade -y
$ sudo apt install -y curl wget gnupg lsb-release ca-certificates

Defina o hostname corretamente

O Rundeck usa o nome do host em várias configurações internas (geração de URLs em e-mails, webhooks, API). Configure-o antes de iniciar:

$ sudo hostnamectl set-hostname rundeck.exemplo.com.br
$ echo "127.0.0.1 rundeck.exemplo.com.br rundeck" | sudo tee -a /etc/hosts
§ 03 — Runtime Java

Instalação do Java (JDK 17)

O Rundeck 5.x requer Java 11 ou 17. Use o OpenJDK 17 — é a versão LTS com melhor suporte nos repositórios padrão do Ubuntu e a recomendada pelo projeto.

$ sudo apt install -y openjdk-17-jre-headless
$ java -version

A saída esperada é algo como:

openjdk version "17.0.14" 2025-01-21
OpenJDK Runtime Environment (build 17.0.14+7-Ubuntu-124.04)
OpenJDK 64-Bit Server VM (build 17.0.14+7-Ubuntu-124.04, mixed mode, sharing)
Boa prática

Use a versão -headless. O Rundeck roda sem interface gráfica e a variante headless economiza espaço em disco e superfície de ataque (não instala bibliotecas X11 desnecessárias).

§ 04 — Repositório

Adicionar o repositório oficial

Evite baixar o .deb solto do site — usar o repositório APT oficial garante atualizações automáticas e verificação de assinatura GPG. O projeto mantém um script de bootstrap no GitHub:

$ curl -L https://raw.githubusercontent.com/rundeck/packaging/main/scripts/deb-setup.sh 2>/dev/null | sudo bash -s rundeck
Atenção — curl | bash

Fazer curl | bash em scripts de terceiros é uma prática que merece inspeção. Antes de rodar, abra a URL no navegador e leia o conteúdo. Uma alternativa é fazer a configuração manual do repositório:

Configuração manual do repositório (alternativa)

# Importa a chave GPG
$ curl -fsSL https://packages.rundeck.com/pagerduty/rundeck/gpgkey \
    | sudo gpg --dearmor -o /usr/share/keyrings/rundeck-archive-keyring.gpg

# Adiciona o repositório
$ echo "deb [signed-by=/usr/share/keyrings/rundeck-archive-keyring.gpg] \
    https://packages.rundeck.com/pagerduty/rundeck/any/ any main" \
    | sudo tee /etc/apt/sources.list.d/rundeck.list

$ sudo apt update
§ 05 — Instalação

Instalar o pacote Rundeck

$ sudo apt install -y rundeck

O instalador cria:

  • Usuário de sistema rundeck (sem shell login).
  • Diretório de configuração em /etc/rundeck/.
  • Diretório de dados em /var/lib/rundeck/.
  • Diretório de logs em /var/log/rundeck/.
  • Serviço systemd rundeckd.

Estrutura de diretórios relevantes

/etc/rundeck/
├── rundeck-config.properties   # config geral (server URL, sessão, DB)
├── framework.properties        # identidade do servidor, SSH defaults
├── realm.properties            # usuários locais e senhas
├── jaas-loginmodule.conf       # módulos de autenticação (JAAS)
├── profile                     # variáveis de ambiente e opções JVM
└── ssl/                        # certificados (se TLS interno for usado)
§ 06 — Configuração base

Configuração base

Duas propriedades precisam ser ajustadas antes do primeiro start: o hostname/URL do servidor e o bind address. Isso evita que o Rundeck tente gerar links com localhost e fique exposto em todas as interfaces de rede.

Editar /etc/rundeck/rundeck-config.properties

# URL pública — será usada em e-mails, webhooks, API.
# Se houver Nginx na frente, aponte para a URL pública HTTPS.
grails.serverURL=https://rundeck.exemplo.com.br

# Tempo de sessão (segundos). 1800 = 30 min, recomendado.
server.servlet.session.timeout=1800

# Forçar leitura dos headers X-Forwarded-* (essencial atrás de proxy)
server.use-forward-headers=true

# Usar H2 apenas para testes. Em produção, configure PostgreSQL/MySQL (seção 07).

Editar /etc/rundeck/framework.properties

framework.server.name     = rundeck.exemplo.com.br
framework.server.hostname = rundeck.exemplo.com.br
framework.server.port     = 4440
framework.server.url      = https://rundeck.exemplo.com.br

# Caminho padrão da chave SSH usada para executar em nós remotos
framework.ssh.keypath   = /var/lib/rundeck/.ssh/id_rsa
framework.ssh.user      = rundeck
framework.ssh.timeout   = 0

Fazer o Jetty escutar apenas em localhost

Por padrão o Rundeck escuta em 0.0.0.0:4440. Com reverse proxy, o correto é restringir ao loopback. Edite /etc/rundeck/profile:

# Localize a linha com RDECK_JVM_SETTINGS e adicione:
export RDECK_JVM_OPTS="$RDECK_JVM_OPTS -Dserver.address=127.0.0.1"

Iniciar o serviço

$ sudo systemctl daemon-reload
$ sudo systemctl enable --now rundeckd
$ sudo systemctl status rundeckd
Paciência no primeiro boot

O Rundeck leva entre 30 e 90 segundos para inicializar no primeiro start. Acompanhe o log até ver a mensagem Started Application:

$ sudo tail -f /var/log/rundeck/service.log

Após esse ponto, teste o acesso local:

$ curl -I http://127.0.0.1:4440/
§ 07 — Banco de dados

Migrar para um banco externo

O banco H2 embutido é inadequado para produção: não é atômico em caso de crash, corrompe com facilidade e impossibilita cluster. Para uma instalação séria, use PostgreSQL (recomendado) ou MySQL/MariaDB.

Instalar PostgreSQL

$ sudo apt install -y postgresql postgresql-contrib
$ sudo systemctl enable --now postgresql

Criar banco e usuário

$ sudo -u postgres psql
CREATE USER rundeck WITH PASSWORD 'TROQUE_POR_SENHA_FORTE';
CREATE DATABASE rundeck OWNER rundeck ENCODING 'UTF8';
GRANT ALL PRIVILEGES ON DATABASE rundeck TO rundeck;
\q
Gerar senha forte

Nunca invente senhas de cabeça. Use: openssl rand -base64 32 ou pwgen -s 32 1. Armazene a senha gerada no seu cofre de segredos (Vault, 1Password, Bitwarden).

Apontar o Rundeck para o PostgreSQL

Edite /etc/rundeck/rundeck-config.properties e substitua as linhas do datasource pelas abaixo:

# Remova ou comente qualquer configuração H2 anterior.

dataSource.driverClassName = org.postgresql.Driver
dataSource.url             = jdbc:postgresql://localhost:5432/rundeck
dataSource.username        = rundeck
dataSource.password        = TROQUE_POR_SENHA_FORTE
dataSource.dialect         = org.hibernate.dialect.PostgreSQLDialect

Reiniciar e verificar a criação de schema

$ sudo systemctl restart rundeckd
$ sudo -u postgres psql -d rundeck -c "\dt"

Você deve ver dezenas de tabelas (execution, scheduled_execution, auth_token, etc.). Se aparecerem, o Rundeck conectou com sucesso e criou o schema automaticamente.

Migração de dados existentes

Se você já tinha o Rundeck rodando com H2 e está migrando, os jobs não serão migrados automaticamente. Exporte os projetos antes (rd projects archives export), troque o banco, e reimporte (rd projects archives import).

§ 08 — Endurecimento

Endurecimento de segurança

1

Trocar a senha padrão admin

A instalação vem com admin:admin. Isso é a primeira coisa a trocar — literalmente antes de expor a porta a qualquer rede. Edite /etc/rundeck/realm.properties:

# Formato: usuario: senha_hash,role1,role2
# Gere hash com: echo -n "minha_senha_forte" | md5sum
admin: MD5:5f4dcc3b5aa765d61d8327deb882cf99,user,admin,architect,deploy,build

Melhor ainda: use OBF: (Jetty obfuscation) ou CRYPT: em vez de MD5. Para hashes modernos, migre para LDAP (seção 10).

2

Desabilitar o usuário admin padrão se usar LDAP

Se você configurou LDAP, comente a linha admin: em realm.properties. Manter usuários locais junto com LDAP abre uma porta de fuga perigosa se alguém descobrir a senha local.

3

Ajustar permissões dos arquivos de configuração

$ sudo chown -R rundeck:rundeck /etc/rundeck/
$ sudo chmod 640 /etc/rundeck/*.properties
$ sudo chmod 640 /etc/rundeck/realm.properties
$ sudo chmod 700 /var/lib/rundeck/.ssh/
$ sudo chmod 600 /var/lib/rundeck/.ssh/id_rsa
4

Criar o par SSH para execução remota

O Rundeck executa comandos em nós remotos via SSH. Gere um par dedicado ao Rundeck — nunca reutilize chaves humanas:

$ sudo -u rundeck mkdir -p /var/lib/rundeck/.ssh
$ sudo -u rundeck ssh-keygen -t ed25519 -N "" \
    -f /var/lib/rundeck/.ssh/id_ed25519 \
    -C "rundeck@rundeck.exemplo.com.br"

Distribua a chave pública (.ssh/id_ed25519.pub) para a authorized_keys de um usuário dedicado (ex.: rundeck-agent) nos nós alvo — nunca use root direto, nunca reuse com deploys CI/CD.

5

Política de ACL fechada por padrão

O Rundeck aplica políticas em arquivos .aclpolicy em /etc/rundeck/. O padrão deixa admin com tudo liberado — para qualquer outro perfil, crie políticas explícitas que concedam permissão. Nunca use allow: '*' fora do role admin.

6

Desabilitar execução inline de scripts não assinados

Em /etc/rundeck/framework.properties:

# Bloqueia jobs que tentam executar scripts não aprovados
rundeck.feature.reauthorizeScheduledJobs.enabled = true
rundeck.feature.enhancedNodes.enabled = true
§ 09 — Reverse proxy

Nginx com TLS na frente

Expor o Jetty (servidor embutido do Rundeck) diretamente à internet é desaconselhado. Use Nginx para terminar TLS, aplicar rate limiting e separar responsabilidades.

Instalar Nginx e Certbot

$ sudo apt install -y nginx certbot python3-certbot-nginx

Configuração do site

Crie /etc/nginx/sites-available/rundeck.conf:

# Redireciona HTTP -> HTTPS
server {
    listen 80;
    server_name rundeck.exemplo.com.br;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name rundeck.exemplo.com.br;

    # Certificados gerenciados pelo Certbot (preenchidos no próximo passo)
    ssl_certificate     /etc/letsencrypt/live/rundeck.exemplo.com.br/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/rundeck.exemplo.com.br/privkey.pem;

    # Políticas TLS modernas
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # Headers de segurança
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options nosniff always;
    add_header X-Frame-Options SAMEORIGIN always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Tamanho de uploads (jobs com arquivos anexos)
    client_max_body_size 50M;

    access_log /var/log/nginx/rundeck.access.log;
    error_log  /var/log/nginx/rundeck.error.log;

    location / {
        proxy_pass http://127.0.0.1:4440;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-Host  $host;

        # Jobs longos precisam de timeout alto
        proxy_read_timeout 900s;
        proxy_send_timeout 900s;
        proxy_buffering off;
    }
}

Ativar o site e emitir o certificado

$ sudo ln -s /etc/nginx/sites-available/rundeck.conf /etc/nginx/sites-enabled/
$ sudo nginx -t

# Emite certificado Let's Encrypt válido por 90 dias (renovação automática)
$ sudo certbot --nginx -d rundeck.exemplo.com.br \
    --redirect --agree-tos -m admin@exemplo.com.br

$ sudo systemctl reload nginx
Teste de renovação automática

O Certbot configura um timer systemd que renova automaticamente. Valide com:

sudo certbot renew --dry-run

Atenção ao header X-Forwarded-Proto

Sem server.use-forward-headers=true no Rundeck (configurado na seção 6), o Jetty ignora o header e tenta redirecionar HTTPS de volta para HTTP — quebrando login e causando loops 302. Se o login falhar após subir o proxy, a causa quase sempre é essa.

§ 10 — Autenticação

Autenticação LDAP/AD

Senhas em arquivo flat (realm.properties) funcionam para 2-3 pessoas. Para uma equipe real, integre com LDAP ou Active Directory. Assim, o offboarding de pessoas acontece automaticamente quando a conta corporativa é desativada.

Editar /etc/rundeck/jaas-loginmodule.conf

RDpropertyfilelogin {
    org.eclipse.jetty.jaas.spi.LdapLoginModule required
    debug="false"
    contextFactory="com.sun.jndi.ldap.LdapCtxFactory"
    hostname="ldap.exemplo.com.br"
    port="636"
    bindDn="cn=rundeck-bind,ou=serviceaccounts,dc=exemplo,dc=com,dc=br"
    bindPassword="SENHA_DA_CONTA_DE_BIND"
    authenticationMethod="simple"
    forceBindingLogin="true"
    userBaseDn="ou=people,dc=exemplo,dc=com,dc=br"
    userRdnAttribute="uid"
    userIdAttribute="uid"
    userPasswordAttribute="userPassword"
    userObjectClass="inetOrgPerson"
    roleBaseDn="ou=groups,dc=exemplo,dc=com,dc=br"
    roleNameAttribute="cn"
    roleMemberAttribute="member"
    roleObjectClass="groupOfNames"
    cacheDurationMillis="300000"
    reportStatistics="true";
};
Use sempre LDAPS (porta 636)

LDAP em porta 389 transmite a senha do usuário em claro. Em qualquer ambiente que não seja um laboratório isolado, use LDAPS ou StartTLS. Importe o CA corporativo no truststore do Java se for certificado auto-assinado.

Ativar o módulo no /etc/rundeck/profile

export RDECK_JVM_SETTINGS="-Djava.security.auth.login.config=/etc/rundeck/jaas-loginmodule.conf \
    -Dloginmodule.name=RDpropertyfilelogin \
    -Drundeck.jaaslogin=true"

Reinicie o serviço e teste com uma conta do AD. Usuários que pertencem aos grupos LDAP mapeados nos .aclpolicy terão as permissões correspondentes.

§ 11 — Firewall

Firewall com UFW

Após o Nginx estar encerrando TLS, a porta 4440 nunca deve ser acessível de fora. Configure o UFW para aceitar apenas SSH, HTTP e HTTPS:

$ sudo ufw default deny incoming
$ sudo ufw default allow outgoing
$ sudo ufw allow OpenSSH
$ sudo ufw allow 'Nginx Full'
$ sudo ufw enable
$ sudo ufw status verbose

A porta 4440 continua aberta internamente (Nginx acessa via 127.0.0.1), mas fica invisível de fora. Confirme com um scanner externo:

$ nmap -p 22,80,443,4440 rundeck.exemplo.com.br

Resultado esperado: 22, 80 e 443 open, e 4440 filtered ou closed.

§ 12 — Operação

Backup e monitoramento

O que precisa de backup

  1. Banco de dados PostgreSQL — jobs, execuções, tokens, webhooks.
  2. /etc/rundeck/ — toda a configuração.
  3. /var/lib/rundeck/projects/ — definições de projetos e SCM local.
  4. /var/lib/rundeck/logs/rundeck/ — logs de execução (se auditoria longa for exigida).

Script simples de backup

#!/bin/bash
# /usr/local/bin/rundeck-backup.sh
set -euo pipefail

BACKUP_DIR="/var/backups/rundeck"
DATE=$(date +%Y%m%d-%H%M)
mkdir -p "$BACKUP_DIR"

# Dump do banco
sudo -u postgres pg_dump rundeck | gzip > "$BACKUP_DIR/db-$DATE.sql.gz"

# Tarball da configuração e projetos
tar czf "$BACKUP_DIR/config-$DATE.tar.gz" \
    /etc/rundeck \
    /var/lib/rundeck/projects

# Retenção: mantém 14 dias
find "$BACKUP_DIR" -type f -name "*.gz" -mtime +14 -delete

Agende via cron (ou, melhor ainda, um timer systemd):

$ sudo chmod +x /usr/local/bin/rundeck-backup.sh
$ echo "0 3 * * * root /usr/local/bin/rundeck-backup.sh" | sudo tee /etc/cron.d/rundeck-backup

Monitoramento essencial

Os sinais mais importantes para alertar:

SinalFerramenta / EndpointAlerta se…
Serviço vivo systemctl is-active rundeckd retorno ≠ active
Healthcheck HTTP GET /health via proxy status ≠ 200 por > 2 min
Heap JVM JMX ou endpoint /metrics/metrics uso > 85% sustentado
Jobs em falha Webhook → Slack/Teams falha consecutiva ≥ 2
Certificado TLS certbot certificates expira em < 20 dias
§ 13 — Fechamento

Checklist de produção

Antes de declarar a instalação "pronta", confirme cada item abaixo:

  • Senha padrão admin:admin removida ou trocada.
  • Banco de dados PostgreSQL configurado (não H2).
  • TLS válido (Let's Encrypt) com redirect HTTP→HTTPS.
  • Porta 4440 inacessível externamente (UFW).
  • server.use-forward-headers=true configurado (evita loop de login).
  • SSH key dedicada, em ed25519, com permissões 600.
  • LDAP/AD integrado (se aplicável) com LDAPS.
  • Backup diário do banco + /etc/rundeck + projetos, com retenção definida.
  • ACLs revisadas — ninguém tem * fora do papel admin.
  • Monitoramento com alerta de serviço e certificado.
  • Log rotation ativa em /var/log/rundeck/ (verifique logrotate.d).
  • Plano de atualização documentado (Rundeck lança updates mensais).
Próximos passos

Com a base sólida, as próximas frentes que costumam trazer retorno rápido são: (a) habilitar Key Storage do Rundeck para secrets em vez de variáveis em texto; (b) integrar SCM (Git) para versionar os jobs em YAML; (c) configurar webhooks para notificar Slack/Teams em execuções críticas; e (d) estudar o modo Cluster se a equipe crescer.

Uma boa instalação de Rundeck não se mede no primeiro login, mas em seis meses — quando ninguém na equipe lembra mais de onde tiveram que consultar uma senha, e todos os jobs rodam, auditados, sem sobressaltos.
