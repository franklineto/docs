01 — FundamentosO que é o Rundeck e quando usar

O Rundeck é uma plataforma open-source de runbook automation — ela transforma scripts, comandos ad-hoc e procedimentos manuais em jobs com interface web, controle de acesso, histórico de execução e API. É mantido pela PagerDuty e disponível em duas versões: a Community (gratuita, open-source Apache 2.0) e a Enterprise / PagerDuty Runbook Automation (comercial).

O problema que o Rundeck resolve

Todo time de infraestrutura acumula "scripts que só o Zé sabe rodar". Reiniciar um serviço, aplicar um patch, restaurar um backup, limpar cache — tarefas que vivem em shell history, wikis desatualizadas e bookmarks de terminal. O Rundeck centraliza essas operações, permitindo:

  • Self-service controlado: desenvolvedores rodam tarefas operacionais sem precisar de acesso SSH direto aos servidores.
  • Auditoria: cada execução fica registrada com quem, quando, em quais hosts e qual foi a saída.
  • Padronização: mesmo playbook de deploy, mesmo playbook de rollback — sem variação por pessoa.
  • Integração: orquestra ferramentas que já existem (Ansible, Puppet, Chef, Jenkins, Kubernetes) em vez de substituí-las.

Quando Rundeck é a escolha certa

CenárioRundeck resolve?
Delegar operações a devs sem dar SSHSim — caso de uso principal
Runbooks de incident response com um cliqueSim — histórico e RBAC prontos
Automação scheduled (cron centralizado)Sim — com interface e logs
Pipeline CI/CD completoNão — use Jenkins, GitLab CI, GitHub Actions
Gerenciamento de configuração declarativoNão — use Ansible/Puppet/Chef (e chame do Rundeck)
Infrastructure as CodeNão — use Terraform (e chame do Rundeck)
Mantra do Rundeck "Rundeck não substitui suas ferramentas de automação — ele as expõe de forma segura e auditável para os times que precisam usá-las." Se você está escrevendo lógica de automação DENTRO do Rundeck, provavelmente deveria escrever em Ansible e chamar do Rundeck.

02 — SetupInstalação e configuração inicial

Vamos instalar o Rundeck Community em um servidor Ubuntu 24.04. O processo é similar em Debian/RHEL/CentOS, com adaptações menores no gerenciador de pacotes.

Pré-requisitos

Instalar o Java (OpenJDK 17)

O Rundeck 5.x requer Java 17. Em servidores antigos rodando Rundeck 4.x, o Java 11 ainda é aceito.

sudo apt update
sudo apt install -y openjdk-17-jre-headless
java -version
# openjdk version "17.0.x"

Instalação

Adicionar o repositório oficial e instalar
# Adicionar a chave GPG e o repositorio
curl -L https://packages.rundeck.com/pagerduty/rundeck/cfg/gpg/gpg.E881C6C0FCF2DE93.key | \
  sudo apt-key add -
echo "deb https://packages.rundeck.com/pagerduty/rundeck/any/ any main" | \
  sudo tee /etc/apt/sources.list.d/rundeck.list

sudo apt update
sudo apt install -y rundeck

Configuração básica

Definir a URL pública do servidor

Edite /etc/rundeck/rundeck-config.properties e ajuste o grails.serverURL para o endereço pelo qual os usuários acessarão o Rundeck. É crítico: links em e-mails, webhooks e API respondem com base nessa URL. 

# /etc/rundeck/rundeck-config.properties
grails.serverURL=https://rundeck.suaempresa.com.br
dataSource.dbCreate=update
dataSource.url=jdbc:h2:file:/var/lib/rundeck/data/rundeckdb;MVCC=true
# Em producao, troque H2 por MySQL ou PostgreSQL (ver callout abaixo)

Banco de dados em produção

Migrar para PostgreSQL

O H2 é apenas para dev/teste. Em produção, use Postgres ou MySQL.

# /etc/rundeck/rundeck-config.properties
dataSource.url=jdbc:postgresql://localhost:5432/rundeck
dataSource.driverClassName=org.postgresql.Driver
dataSource.username=rundeck
dataSource.password=senha_forte_aqui
dataSource.properties.validationQuery=SELECT 1

Iniciar o serviço

Habilitar e verificar
sudo systemctl enable --now rundeckd
sudo systemctl status rundeckd

# Acompanhar logs em tempo real durante o primeiro boot
sudo tail -f /var/log/rundeck/service.log

# Acessar em http://<seu-servidor>:4440
# Login padrao: admin / admin  ← TROQUE IMEDIATAMENTE

Proxy reverso HTTPS

Colocar atrás do nginx

Nunca exponha o Rundeck direto na porta 4440 em produção. Use nginx ou Caddy com TLS:

# /etc/nginx/sites-available/rundeck
server {
    listen 443 ssl http2;
    server_name rundeck.suaempresa.com.br;

    ssl_certificate     /etc/letsencrypt/live/rundeck.suaempresa.com.br/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/rundeck.suaempresa.com.br/privkey.pem;

    client_max_body_size 50M;

    location / {
        proxy_pass http://127.0.0.1:4440;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 600s;  # jobs longos
    }
}

No framework.properties, avise o Rundeck que está atrás de proxy: framework.server.url=https://rundeck.suaempresa.com.br.

Troque a senha padrão O usuário admin/admin é o primeiro vetor de ataque. Edite /etc/rundeck/realm.properties imediatamente após o boot, use hashes bcrypt (nunca senha em texto plano em produção) ou, melhor ainda, integre com LDAP/AD/SSO.

03 — ArquiteturaComponentes internos do Rundeck

Entender como o Rundeck é organizado por dentro ajuda a diagnosticar problemas e planejar operações de backup e upgrade.

Camadas

  • Servidor web (Grails/Spring Boot): aplicação Java que serve a UI, a API REST e processa as execuções. Escuta na 4440 (HTTP) ou 4443 (HTTPS) por padrão.
  • Banco de dados: armazena projetos, jobs, execuções, logs de auditoria e metadados. H2 embarcado no dev, Postgres/MySQL em produção.
  • Key Storage: repositório criptografado de segredos (chaves SSH, senhas, tokens de API) armazenado em disco em /var/lib/rundeck/var/storage/.
  • Log Storage: saída de cada execução, armazenada em arquivos em /var/lib/rundeck/logs/rundeck/ por padrão (pode ser movido para S3, Azure Blob etc. via plugin).
  • Executores (SSH/WinRM): como o Rundeck efetivamente roda comandos nos nodes. SSH é o padrão para Linux; WinRM para Windows.

Estrutura de diretórios

/etc/rundeck/              # Configuracao
  ├── framework.properties    # Config do framework (paths, URL)
  ├── rundeck-config.properties  # Config da aplicacao (DB, email)
  ├── realm.properties        # Usuarios locais e senhas
  ├── profile                 # JAVA_OPTS, variaveis do servico
  └── *.aclpolicy             # Politicas de autorizacao

/var/lib/rundeck/
  ├── data/                   # H2 database (se em uso)
  ├── logs/                   # Logs de execucao dos jobs
  ├── projects/               # Definicoes YAML dos projetos
  │   └── <nome-projeto>/
  │       ├── etc/            # project.properties, node sources
  │       └── var/            # cache e state do projeto
  └── var/storage/            # Key Storage (segredos criptografados)

/var/log/rundeck/
  ├── service.log             # Log principal do servidor
  ├── rundeck.audit.log       # Auditoria de autorizacao
  └── rundeck.access.log      # Access log tipo Apache

Fluxo de uma execução

  1. Usuário clica "Run Job" na UI (ou faz POST via API).
  2. Rundeck verifica permissões no ACL correspondente ao grupo do usuário.
  3. Se autorizado, resolve o node filter — descobre em quais hosts rodar.
  4. Para cada node, abre conexão SSH/WinRM e executa os passos do workflow.
  5. Saída é streamada para o Log Storage; execução é registrada no banco.
  6. Notificações (e-mail, webhook, Slack) são disparadas se configuradas.

04 — SegurançaAutenticação e autorização (ACLs)

O modelo de segurança do Rundeck tem duas camadas distintas: autenticação (quem é você) e autorização (o que você pode fazer). Cada uma é configurada separadamente.

Autenticação — quem é o usuário

O Rundeck suporta várias fontes de identidade, configuráveis via JAAS (arquivo jaas-loginmodule.conf):

  • Realm properties file: usuários em arquivo texto — bom para dev/teste, ruim para produção.
  • LDAP / Active Directory: integração com diretório corporativo.
  • PAM: delegação para usuários do sistema operacional.
  • SSO via SAML/OIDC: disponível na versão Enterprise (Okta, Azure AD, Google Workspace).

Arquivo realm.properties para dev

# /etc/rundeck/realm.properties
# Formato: usuario: senha[,hash], grupo1, grupo2...
admin:   MD5:21232f297a57a5a743894a0e4a801fc3, admin, user
jose:    $2a$10$Lr5K...bcrypt_hash..., dev, user
maria:   $2a$10$Mn7P...bcrypt_hash..., devops, user
api_bot: $2a$10$Qr9X...bcrypt_hash..., api

Integração com LDAP (produção)

# /etc/rundeck/jaas-loginmodule.conf
RDpropertyfileloginmodule {
  org.rundeck.jaas.jetty.JettyCachingLdapLoginModule required
  debug="true"
  contextFactory="com.sun.jndi.ldap.LdapCtxFactory"
  providerUrl="ldaps://ad.empresa.com.br:636"
  bindDn="CN=rundeck-svc,OU=Service,DC=empresa,DC=com,DC=br"
  bindPassword="senha_servico"
  authenticationMethod="simple"
  forceBindingLogin="true"
  userBaseDn="OU=Users,DC=empresa,DC=com,DC=br"
  userRdnAttribute="sAMAccountName"
  userIdAttribute="sAMAccountName"
  userPasswordAttribute="unicodePwd"
  userObjectClass="user"
  roleBaseDn="OU=Groups,DC=empresa,DC=com,DC=br"
  roleNameAttribute="cn"
  roleMemberAttribute="member"
  roleObjectClass="group"
  cacheDurationMillis="300000"
  nestedGroups="true"
  reportStatistics="true";
};

Autorização — o que o usuário pode fazer

Toda autorização é definida em arquivos .aclpolicy no diretório /etc/rundeck/. São YAMLs que descrevem regras do tipo "grupo X pode fazer ação Y no recurso Z dentro do contexto W".

Estrutura de uma ACL

# /etc/rundeck/devs.aclpolicy
description: Desenvolvedores - acesso ao projeto MeuApp
context:
  project: MeuApp        # contexto: projeto especifico
for:
  job:
    - allow: [read, run]  # ver e executar qualquer job
  node:
    - allow: [read, run]
  adhoc:
    - deny: [run, run_as_user]  # proibir comandos ad-hoc
by:
  group: [devs]
---
# Segundo documento YAML no mesmo arquivo (--- separa)
description: Devs - acesso a listar o projeto
context:
  application: rundeck   # contexto: aplicacao (fora de projeto)
for:
  project:
    - equals:
        name: MeuApp
      allow: [read]
by:
  group: [devs]

Conceitos-chave da ACL

  • Contexto: application: rundeck (nível de sistema) ou project: nome (dentro de um projeto). Ambos são necessários para dar acesso completo — sem o contexto de aplicação, o usuário nem vê o projeto na lista.
  • Default deny: se nada autoriza explicitamente uma ação, ela é negada. Um deny explícito tem precedência sobre qualquer allow.
  • Matching: equals (match exato), match (regex), ou contains (para listas).
  • Hot reload: mudanças em .aclpolicy são detectadas em até 2 minutos sem restart.

Exemplo completo: política por ambiente

# Devs podem rodar qualquer coisa em dev, so leitura em prod
description: Dev team - full access dev, read-only prod
context:
  application: rundeck
for:
  project:
    - match:
        name: '.*-dev|.*-staging'
      allow: [read, configure]
    - match:
        name: '.*-prod'
      allow: [read]
by:
  group: [dev-team]
---
description: Dev team - job permissions per env
context:
  project: '.*-dev|.*-staging'
for:
  job:
    - allow: [read, run, create, update, delete, kill, view]
  node:
    - allow: [read, run]
  adhoc:
    - allow: [read, run]
  resource:
    - allow: [read, create, update, delete]
  storage:
    - allow: [read]
by:
  group: [dev-team]
---
description: Dev team - prod read-only
context:
  project: '.*-prod'
for:
  job:
    - allow: [read, view]
  node:
    - allow: [read]
by:
  group: [dev-team]
Ferramenta essencial: rd acl O CLI rd acl test permite simular uma decisão de autorização sem precisar logar na UI. Exemplo: rd acl test --context project:MeuApp --groups devs --job name:deploy --allow run. Se retornar REJECTED, ele mostra a regra que faltaria para autorizar. Insubstituível para debugar ACLs complexas.

05 — ProjetosProjetos, nodes e fontes de inventário

Um projeto é a unidade de isolamento no Rundeck: tem seus próprios jobs, nodes, ACLs e configurações. Times diferentes geralmente têm projetos separados, ou projetos por ambiente (app-dev, app-prod).

Criar um projeto

# Via CLI (rd)
rd projects create -p MeuApp \
  --property "project.description=Deploy e operacao do MeuApp" \
  --property "service.NodeExecutor.default.provider=sshj-ssh" \
  --property "service.FileCopier.default.provider=sshj-scp"

# Ou via API
curl -X POST "$RD_URL/api/48/projects" \
  -H "X-Rundeck-Auth-Token: $RD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"MeuApp","description":"Deploy e operacao do MeuApp"}'

Node sources — de onde vem a lista de servidores

Nodes no Rundeck podem vir de várias fontes, misturadas no mesmo projeto:

1. Arquivo YAML estático (local)

# /var/lib/rundeck/projects/MeuApp/etc/nodes.yaml
web01:
  hostname: 10.0.1.10
  username: rundeck
  tags: web,prod,app-frontend
  osFamily: unix
  osName: Linux
  ssh-authentication: privateKey
  ssh-key-storage-path: keys/prod/deploy.key

web02:
  hostname: 10.0.1.11
  username: rundeck
  tags: web,prod,app-frontend
  osFamily: unix

db01:
  hostname: 10.0.2.10
  username: rundeck
  tags: database,prod,postgres
  osFamily: unix

2. AWS EC2 (descoberta dinâmica)

Com o plugin de EC2, o Rundeck consulta a API da AWS a cada N segundos e popula nodes automaticamente. Útil em ambientes auto-escaláveis: 

# project.properties
resources.source.1.type=aws-ec2
resources.source.1.config.region=us-east-1
resources.source.1.config.filter=tag:Environment=production
resources.source.1.config.mappingParams=tags.selector=tags/Role
resources.source.1.config.refreshInterval=60

3. URL externa (CMDB, Consul, Prometheus)

resources.source.2.type=url
resources.source.2.config.url=https://cmdb.empresa.com.br/api/nodes.yaml
resources.source.2.config.headers.Authorization=Bearer ${token}
resources.source.2.config.cache=true
resources.source.2.config.refreshInterval=300

Node filters — selecionar subconjuntos

Um node filter é uma expressão que seleciona nodes para rodar um job. Sintaxe baseada em atributo: valor: 

# Todos os nodes com a tag "web" em producao
tags: web+prod

# Web em producao, exceto Windows
tags: web+prod !os-family: windows

# Hostname comeca com "db-" e tem tag "replica"
name: db-.* tags: replica

# OR entre tags (vírgula)
tags: web,api
Operadores + significa AND, , significa OR, e o prefixo ! nega. tags: web+prod é "web E prod"; tags: web,api é "web OU api". Atributos suportam regex: name: prod-web-\d+.

06 — OperaçãoLogs, backup e troubleshooting

Logs — o que procurar onde

ArquivoServe para
/var/log/rundeck/service.logErros da aplicação, stack traces, problemas de inicialização
/var/log/rundeck/rundeck.audit.logDecisões de autorização (ACL) — essencial para debugar permissões
/var/log/rundeck/rundeck.access.logRequisições HTTP (UI e API) — formato Apache
/var/lib/rundeck/logs/rundeck/<projeto>/job/...Output de cada execução de job, por UUID

Backup mínimo recomendado

#!/usr/bin/env bash
# Backup completo do Rundeck
set -euo pipefail

BACKUP_DIR="/backups/rundeck/$(date +%Y%m%d-%H%M)"
mkdir -p "$BACKUP_DIR"

# 1. Configuracao
tar czf "$BACKUP_DIR/etc.tar.gz" /etc/rundeck/

# 2. Projetos, key storage, logs
tar czf "$BACKUP_DIR/var-lib.tar.gz" \
  /var/lib/rundeck/projects \
  /var/lib/rundeck/var/storage \
  /var/lib/rundeck/logs

# 3. Banco de dados (Postgres)
sudo -u postgres pg_dump rundeck | gzip > "$BACKUP_DIR/rundeck-db.sql.gz"

# 4. Exportar todos os projetos via API (arquivo .rdproject)
for project in $(rd projects list --outformat '%name'); do
  rd projects archives export -p "$project" \
    -f "$BACKUP_DIR/project-${project}.rdproject"
done

# Subir para S3/B2
aws s3 sync "$BACKUP_DIR" "s3://backups/rundeck/$(date +%Y%m%d-%H%M)/"

Troubleshooting — problemas comuns

Job falha com "permission denied" no SSH

O usuário rundeck do processo precisa ter sua chave pública autorizada no ~/.ssh/authorized_keys do usuário remoto. A chave privada fica no Key Storage (keys/...) referenciada no projeto. Verifique com sudo -u rundeck ssh usuario@host diretamente.

ACL parece correta mas o usuário não consegue acessar

Em 90% dos casos, falta a regra de contexto application: rundeck dando read ao projeto. ACLs só de projeto não funcionam sem isso. Confirme com tail -f /var/log/rundeck/rundeck.audit.log durante a tentativa de acesso — ele mostra exatamente qual recurso e ação foi rejeitada.

Rundeck lento após meses de uso

A tabela de execuções cresce sem limite. Configure rundeck.execution.logs.fileStorage.cleanerPolicyJob e rundeck.execution.logs.fileStorage.retentionPeriod para arquivar execuções antigas. Típico: manter 90 dias de logs online.

07 — JobsAnatomia de um job YAML

Todo job no Rundeck pode ser criado pela UI, mas o formato canônico é YAML — o que permite versionar em Git, revisar com pull requests e mover entre ambientes via rd jobs load.

O job mais simples possível

- name: hello-world
  description: Um Ola Mundo para testar Rundeck
  loglevel: INFO
  sequence:
    keepgoing: false
    strategy: node-first
    commands:
      - exec: echo "Ola do servidor $(hostname)"
  nodefilters:
    filter: 'tags: web+prod'

Importe via CLI: rd jobs load -p MeuProjeto -f hello.yaml --format yaml.

Estrutura completa explicada

- name: deploy-app                     # Nome visivel na UI
  uuid: 7f3a9b12-1234-5678-9abc-def012345678   # Unico, gerado automaticamente
  group: operacao/deploy                # Agrupamento hierarquico com "/"
  description: |
    Deploy da aplicacao MeuApp.

    Este job para o servico, puxa a versao nova do artifact repo,
    aplica migrations e religa o servico. Rollback automatico se
    o health check falhar.
  loglevel: INFO                        # DEBUG | VERBOSE | INFO | WARN | ERROR
  executionEnabled: true                # Permite execucao (pode desabilitar sem deletar)
  scheduleEnabled: true                 # Permite schedule (cron)
  nodeFilterEditable: false             # Impede override do filter na execucao
  nodesSelectedByDefault: true          # Pre-seleciona nodes ao abrir a tela

  # Options (parametros de entrada)
  options:
    - name: versao
      required: true
      description: "Versao a ser deployada (ex: 1.2.3)"
      regex: '^\d+\.\d+\.\d+$'

    - name: rollback-on-failure
      description: "Fazer rollback automatico se falhar?"
      type: boolean
      value: "true"

  # Seletor de nodes
  nodefilters:
    dispatch:
      threadcount: 3                    # Paralelismo entre nodes
      keepgoing: false                  # Para ao primeiro erro
      rankAttribute: rank
      rankOrder: ascending
      successOnEmptyNodeFilter: false
    filter: 'tags: app-frontend+prod'

  # Agendamento (cron opcional)
  schedule:
    time:
      hour: '02'
      minute: '00'
    weekday:
      day: '*'
    month: '*'
    year: '*'

  # Workflow (os passos)
  sequence:
    keepgoing: false                    # Para na primeira falha
    strategy: node-first                # node-first | step-first | parallel
    commands:
      - description: "1. Parar servico"
        exec: sudo systemctl stop meuapp

      - description: "2. Puxar nova versao"
        script: |
          #!/bin/bash
          set -euo pipefail
          cd /opt/meuapp
          curl -sSL -o app.tar.gz \
            "https://artifacts.empresa.com/meuapp-@option.versao@.tar.gz"
          tar xzf app.tar.gz
        scriptargs: ""

      - description: "3. Rodar migrations"
        exec: /opt/meuapp/bin/migrate.sh
        errorhandler:
          description: "Rollback da migration"
          exec: /opt/meuapp/bin/migrate.sh --rollback
          keepgoingOnSuccess: false

      - description: "4. Subir servico"
        exec: sudo systemctl start meuapp

      - description: "5. Health check"
        exec: curl -f http://localhost:8080/health

  # Notificacoes
  notification:
    onsuccess:
      email:
        recipients: devops@empresa.com.br
        subject: "[OK] Deploy MeuApp ${option.versao}"
    onfailure:
      webhook:
        urls: https://hooks.slack.com/services/AAA/BBB/CCC

  notifyAvgDurationThreshold: "+30"     # Alerta se demorar 30s acima da media

Tipos de passo (steps)

TipoPara quê
execComando shell de uma linha no node remoto
scriptScript inline (bash, python, etc.) enviado e executado no node
scriptfileArquivo de script já presente no node
scripturlScript baixado de uma URL (útil com S3 presigned URLs)
jobrefChama outro job do Rundeck (encadeamento)
nodeStep: false + pluginStep que roda no servidor Rundeck, não nos nodes (ex: Ansible, Docker)

08 — InputsOptions, variáveis e Key Storage

Options — inputs do usuário

Options são os parâmetros que o usuário preenche ao rodar o job. Podem ser texto livre, listas com valores fixos ou valores carregados dinamicamente de um endpoint HTTP.

Option de texto com validação

options:
  - name: hostname
    description: Hostname do servidor a reiniciar
    required: true
    regex: '^[a-z0-9-]+\.empresa\.com\.br$'

Lista fixa (dropdown)

options:
  - name: ambiente
    description: Ambiente alvo
    required: true
    enforced: true                 # usuario nao pode digitar valor livre
    values:
      - dev
      - staging
      - prod
    value: dev                     # default

Lista dinâmica via HTTP (remote options)

options:
  - name: branch
    description: Branch do Git a deployar
    required: true
    enforced: true
    valuesUrl: 'https://gitlab.empresa.com/api/v4/projects/42/repository/branches'
    # O endpoint deve retornar JSON no formato:
    # [{"name":"main","value":"main"},{"name":"release-1.2","value":"release-1.2"}]

Option secreta (senha, mascarada na UI e nos logs)

options:

    description: Senha de admin do banco
    required: true

Referenciando options nos passos

sequence:

Key Storage — segredos criptografados

Nunca coloque senhas ou chaves SSH em texto plano nos jobs YAML. Use o Key Storage: um KV hierárquico criptografado que guarda chaves privadas, passwords e tokens. 

# Armazenar uma chave SSH privada
rd keys create \
  --path "keys/prod/deploy.key" \
  --type privateKey \
  --file ~/.ssh/id_rsa_prod

# Armazenar uma senha
echo -n "senha_super_secreta" | rd keys create \
  --path "keys/prod/db-password" \
  --type password

# Listar chaves (so metadados, nao o conteudo)
rd keys list --path keys/prod/

Usando chaves em um job

# Opcao 1: chave SSH para o node (definida em nodes.yaml)
web01:
  hostname: 10.0.1.10
  ssh-authentication: privateKey
  ssh-key-storage-path: keys/prod/deploy.key

# Opcao 2: exposicao como variavel secreta no job
options:
  - name: db-password
    label: "Senha do banco"
    required: true
    secure: true
    storagePath: keys/prod/db-password      # le do Key Storage
    valueExposed: true                      # disponivel como @option.db-password@

sequence:
  commands:
    - script: |
        PGPASSWORD='@option.db-password@' psql -h db.empresa.com -U app -c "SELECT 1"
Nunca imprima secrets echo $PASSWORD no script vai aparecer no log de execução — e qualquer um com permissão view_log pode ler. Use secure: true e tome cuidado com redirecionamento de stderr.

09 — WorkflowsEstratégias de execução e tratamento de erros

Três estratégias de execução

  • Node-first (padrão): para cada node, roda todos os passos em ordem. Vai para o próximo node só quando o primeiro terminou. Bom para deploys: você para, atualiza e religa um servidor antes de mexer no próximo.
  • Step-first: para cada passo, roda em todos os nodes antes de passar ao próximo passo. Bom para operações paralelas em batch (ex: "parar em todos, depois atualizar em todos, depois ligar em todos").
  • Parallel (plugin): roda passos independentes em paralelo — útil para verificações de saúde simultâneas.

keepgoing — continuar após erro?

sequence:
  keepgoing: true         # Mesmo se um passo falhar, continua os proximos
  strategy: node-first
  commands:
    - exec: systemctl status nginx
    - exec: systemctl status postgres
    - exec: df -h
    - exec: free -m
# Util para jobs de "health check geral" onde cada verificacao e independente

Error handlers — ação se um passo falhar

sequence:
  commands:
    - description: "Aplicar migration"
      exec: /opt/app/bin/migrate.sh up
      errorhandler:
        description: "Reverter migration em caso de falha"
        exec: /opt/app/bin/migrate.sh down
        keepgoingOnSuccess: false    # se o handler sucedeu, ainda assim para o job

Referência entre jobs (jobref)

Chamar um job de dentro de outro é a forma de compor fluxos complexos a partir de peças reutilizáveis. O job pai espera o job filho terminar antes de prosseguir. 

sequence:
  commands:
    - jobref:
        name: validar-pre-deploy
        group: infra/checks
        args: "-env prod -app meuapp"

    - jobref:
        name: deploy-app
        group: operacao/deploy
        args: "-versao ${option.versao}"
        # Opcional: override do filtro de nodes do job filho
        nodefilters:
          filter: 'tags: canary+prod'
          dispatch:
            threadcount: 1
            keepgoing: false

    - jobref:
        name: smoke-tests
        group: qa
        failOnDisable: true       # falha se o job filho estiver desabilitado

Paralelismo controlado com threadcount

threadcount: 5 significa que o Rundeck vai rodar em até 5 nodes simultaneamente. Se você tem 100 web servers, isso evita deploy "em bloco" que tira o site do ar por completo se algo der errado. Combine com keepgoing: false para parar na primeira falha e ter uma "janela de segurança".

10 — Casos de usoOs jobs mais comuns no dia-a-dia

Caso 1 — Reiniciar um serviço com confirmação

BÁSICO · SELF-SERVICE

Cenário Dev precisa reiniciar o serviço da app em staging às vezes — mas sem ter acesso SSH direto, e sem ligar pro time de ops. 

- name: restart-service
  group: self-service
  description: Reinicia um servico systemd em staging
  nodesSelectedByDefault: false
  nodeFilterEditable: true
  options:
    - name: service
      required: true
      enforced: true
      values: [meuapp, worker, scheduler, redis]
    - name: confirmacao
      required: true
      description: 'Digite REINICIAR para confirmar'
      regex: '^REINICIAR$'

  nodefilters:
    dispatch:
      threadcount: 1
      keepgoing: false
    filter: 'tags: app+staging'

  sequence:
    keepgoing: false
    strategy: node-first
    commands:
      - description: 'Parar servico'
        exec: sudo systemctl stop ${option.service}

      - description: 'Aguardar 2s'
        exec: sleep 2

      - description: 'Iniciar servico'
        exec: sudo systemctl start ${option.service}

      - description: 'Verificar status'
        exec: sudo systemctl is-active ${option.service}

Caso 2 — Deploy blue/green com rollback

INTERMEDIÁRIO · DEPLOY

Cenário Deploy da app em produção com paralelismo controlado, health check, e rollback automático se 2 de 10 nodes falharem. 

- name: deploy-rolling
  group: operacao/deploy
  description: Deploy rolling com rollback automatico
  options:
    - name: versao
      required: true
      regex: '^\d+\.\d+\.\d+$'
    - name: paralelismo
      value: '2'
      description: 'Quantos nodes deployar simultaneamente'

  nodefilters:
    dispatch:
      threadcount: ${option.paralelismo}
      keepgoing: false        # Para na primeira falha
    filter: 'tags: app-frontend+prod'

  sequence:
    keepgoing: false
    strategy: node-first      # Importante: node-first para deploy rolling
    commands:
      - description: 'Remover do load balancer'
        exec: |
          curl -X POST "http://lb.empresa.com/drain?node=${node.name}"

      - description: 'Aguardar draining (30s)'
        exec: sleep 30

      - description: 'Baixar artefato'
        script: |
          cd /opt/meuapp
          curl -sSL -o new.tar.gz \
            "https://artifacts.empresa.com/meuapp-@option.versao@.tar.gz"
          tar xzf new.tar.gz -C /opt/meuapp/releases/@option.versao@/

      - description: 'Atualizar symlink e reiniciar'
        exec: |
          sudo ln -sfn /opt/meuapp/releases/@option.versao@ /opt/meuapp/current &&
          sudo systemctl restart meuapp
        errorhandler:
          description: 'Rollback do symlink'
          exec: |
            sudo ln -sfn /opt/meuapp/releases/previous /opt/meuapp/current &&
            sudo systemctl restart meuapp

      - description: 'Health check'
        exec: |
          for i in $(seq 1 30); do
            curl -sf http://localhost:8080/health && exit 0
            sleep 2
          done
          exit 1

      - description: 'Voltar ao load balancer'
        exec: |
          curl -X POST "http://lb.empresa.com/enable?node=${node.name}"

  notification:
    onfailure:
      webhook:
        urls: https://hooks.slack.com/services/AAA/BBB/CCC
    onsuccess:
      email:
        recipients: devops@empresa.com.br
        subject: '[OK] Deploy ${option.versao} concluido'

Por que funciona: threadcount: 2 com keepgoing: false garante que se o node 1 ou 2 falharem, os próximos 8 não são deployados. O error handler na etapa crítica reverte o symlink. O health check ao final valida que o serviço subiu antes de reabilitar no LB.

Caso 3 — Backup diário de banco com cron

INTERMEDIÁRIO · AGENDADO

Cenário Backup noturno de Postgres, com upload para S3 e rotação de backups mais antigos que 30 dias. 

- name: backup-postgres-diario
  group: infra/backups
  description: Backup noturno dos bancos Postgres
  scheduleEnabled: true
  schedule:
    time: { hour: '02', minute: '00', seconds: '0' }
    weekday: { day: '*' }
    month: '*'
    year: '*'

  options:
    - name: retention-days
      value: '30'
      description: 'Dias de retencao'

  nodefilters:
    dispatch:
      threadcount: 1
      keepgoing: true
    filter: 'tags: postgres+primary'

  sequence:
    keepgoing: true
    strategy: node-first
    commands:
      - description: 'Dump do banco'
        script: |
          #!/bin/bash
          set -euo pipefail
          DATE=$(date +%Y%m%d-%H%M)
          FILE="/tmp/pgbackup-${node.name}-$DATE.sql.gz"

          sudo -u postgres pg_dumpall | gzip > "$FILE"
          echo "Backup criado: $FILE ($(du -h $FILE | cut -f1))"

      - description: 'Upload para S3'
        script: |
          DATE=$(date +%Y%m%d-%H%M)
          aws s3 cp "/tmp/pgbackup-${node.name}-$DATE.sql.gz" \
            "s3://backups-empresa/postgres/${node.name}/$DATE.sql.gz" \
            --storage-class STANDARD_IA

          rm "/tmp/pgbackup-${node.name}-$DATE.sql.gz"

      - description: 'Rotacao - remover backups antigos'
        script: |
          aws s3 ls "s3://backups-empresa/postgres/${node.name}/" | \
            awk '{print $4}' | \
            while read obj; do
              # Parse data do nome do arquivo
              obj_date=$(echo "$obj" | grep -oP '\d{8}')
              if [ -n "$obj_date" ]; then
                cutoff=$(date -d "@option.retention-days@ days ago" +%Y%m%d)
                if [ "$obj_date" -lt "$cutoff" ]; then
                  aws s3 rm "s3://backups-empresa/postgres/${node.name}/$obj"
                fi
              fi
            done

  notification:
    onfailure:
      email:
        recipients: dba@empresa.com.br
        subject: '[FALHA] Backup Postgres - ${node.name}'

Caso 4 — Runbook de incidente (high CPU)

AVANÇADO · INCIDENT RESPONSE

Cenário PagerDuty acorda o on-call às 3h da manhã: "High CPU em web-07". Em vez de SSH manual, o on-call roda um runbook que coleta diagnóstico e tenta remediação automática. 

- name: incident-high-cpu
  group: incidents/runbooks
  description: |
    Diagnostica e tenta remediar high CPU em um node.
    Coleta top processes, stack traces do Java e, opcionalmente,
    reinicia o servico se a flag --remediate for true.

  options:
    - name: hostname
      required: true
      description: 'Hostname do node com high CPU'
    - name: remediate
      type: boolean
      value: 'false'
      description: 'Tentar reiniciar o servico apos diagnostico?'

  nodefilters:
    filter: 'name: ${option.hostname}'
    dispatch:
      threadcount: 1
      keepgoing: true

  sequence:
    keepgoing: true           # Continua coletando mesmo que um passo falhe
    strategy: node-first
    commands:
      - description: '1. Top processes'
        exec: ps aux --sort=-%cpu | head -20

      - description: '2. Load average historico'
        exec: uptime; cat /proc/loadavg

      - description: '3. Thread dump (se for Java)'
        script: |
          if pgrep -f java >/dev/null; then
            JAVA_PID=$(pgrep -f "meuapp.jar" | head -1)
            if [ -n "$JAVA_PID" ]; then
              sudo -u app jstack $JAVA_PID > /tmp/threaddump-$(date +%s).txt
              echo "Thread dump salvo em /tmp/"
              tail -200 /tmp/threaddump-*.txt
            fi
          fi

      - description: '4. Ultimos logs do servico'
        exec: sudo journalctl -u meuapp -n 100 --no-pager

      - description: '5. Conexoes de rede'
        exec: sudo ss -tunap | head -40

      - description: '6. Remediacao (se solicitado)'
        script: |
          if [ "@option.remediate@" = "true" ]; then
            echo ">>> Reiniciando servico meuapp"
            sudo systemctl restart meuapp
            sleep 5
            sudo systemctl status meuapp
          else
            echo ">>> Remediacao nao solicitada (flag --remediate false)"
          fi

  notification:
    onsuccess:
      webhook:
        urls: "https://events.pagerduty.com/v2/enqueue"

O valor desse job não é automatizar tudo — é garantir que o on-call colete os mesmos dados a cada incidente, sem esquecer um comando crítico às 3h da manhã. E o histórico de execução vira post-mortem automaticamente.

Caso 5 — Invocar Ansible do Rundeck

INTERMEDIÁRIO · INTEGRAÇÃO

Cenário Time já tem playbooks Ansible maduros. Quer expor a execução deles via Rundeck para o time de dev, sem dar acesso ao repo Ansible. 

- name: executar-playbook-ansible
  group: ansible
  description: Executa um playbook Ansible selecionado
  options:
    - name: playbook
      required: true
      enforced: true
      values:
        - site.yml
        - deploy-app.yml
        - patch-security.yml
        - rotate-certs.yml
    - name: hosts
      required: true
      description: 'Grupo do inventario (ex: web-prod, db-staging)'
      enforced: true
      valuesUrl: 'https://ansible-control/api/groups'
    - name: extra-vars
      description: 'Extra vars JSON (opcional, ex: {"versao":"1.2.3"})'
      value: '{}'

  nodefilters:
    filter: 'name: ansible-control-01'   # Roda no servidor Ansible
    dispatch:
      threadcount: 1

  sequence:
    keepgoing: false
    commands:
      - description: 'Rodar ansible-playbook'
        script: |
          #!/bin/bash
          set -euo pipefail
          cd /opt/ansible

          # Pull do repo para garantir versao mais recente
          git pull origin main

          # Execucao
          ansible-playbook \
            -i inventory/prod \
            --limit "@option.hosts@" \
            --extra-vars '@option.extra-vars@' \
            --extra-vars "rundeck_user=${job.username}" \
            --extra-vars "rundeck_execid=${job.execid}" \
            playbooks/@option.playbook@

Observe os extras rundeck_user e rundeck_execid — o playbook pode usá-los para logar quem disparou e criar links de volta à execução no Rundeck. Rastreabilidade completa.

Caso 6 — Workflow com steps condicionais

AVANÇADO · CONTROLE DE FLUXO

Cenário Job que executa passos diferentes dependendo da hora do dia ou de uma variável de estado — por exemplo, pular migrations em horário comercial. 

- name: deploy-com-condicional
  group: deploys
  description: Deploy que pula migration em horario comercial
  options:
    - name: versao
      required: true

  sequence:
    keepgoing: false
    commands:
      - description: 'Detectar hora e setar flag'
        script: |
          HOUR=$(date +%H)
          if [ "$HOUR" -ge 8 ] && [ "$HOUR" -lt 18 ]; then
            echo "RUNDECK:DATA:skip-migration=true"
          else
            echo "RUNDECK:DATA:skip-migration=false"
          fi

      # O echo RUNDECK:DATA:key=value vira variavel @data.key@
      - description: 'Rodar migration (apenas fora do horario comercial)'
        exec: /opt/app/bin/migrate.sh
        plugins:
          LogFilter:
            - type: key-value-data
              config: { logData: 'true' }
        condition: '${data.skip-migration} == "false"'

      - description: 'Deploy da aplicacao'
        exec: /opt/app/bin/deploy.sh @option.versao@
key-value-data log filter O plugin key-value-data parsea stdout procurando linhas RUNDECK:DATA:chave=valor e injeta como variáveis ${data.chave} nos próximos passos. É o mecanismo oficial para passar estado entre etapas de um workflow.

12 — Boas práticasArmadilhas comuns e como evitá-las

Versione seus jobs em Git

Jobs criados pela UI ficam só no banco do Rundeck. Se o servidor morrer, você perde tudo. Trate jobs como código: exporte para YAML, commite no Git e use rd jobs load num pipeline CI para aplicar no Rundeck (GitOps).

Nomeie grupos com /

O campo group aceita / como separador hierárquico. operacao/deploy/frontend fica organizado na UI como uma árvore navegável. Com centenas de jobs, isso vira essencial.

Idempotência sempre

Assuma que o job vai ser executado duas vezes seguidas — por acidente, por retry, por outro operador. Rode duas vezes o job de deploy e o estado final deve ser o mesmo. systemctl restart é idempotente; mv file file.bak não é (falha na segunda vez).

keepgoing cuidadoso

keepgoing: true é tentador mas perigoso em deploys — um node falha, os outros 50 continuam sendo deployados com problema. Em deploys, sempre keepgoing: false. Em health checks e coletas de diagnóstico, keepgoing: true é apropriado.

Scripts longos merecem arquivo separado

Um bash de 80 linhas embutido no YAML fica ilegível. Use scriptfile: apontando para um arquivo no repo, ou scripturl: para um presigned URL no S3. Ou melhor: chame um playbook Ansible. O Rundeck orquestra, as ferramentas especializadas executam.

Monitore o Rundeck

Exporte métricas do endpoint /metrics (Prometheus format) para Grafana. Acompanhe: tempo médio de execução por job, taxa de falha, tamanho da fila de execuções, uso de memória da JVM. Se o Rundeck morre, seus runbooks morrem junto.

Regra do dedão Se um job roda mais de 15 minutos, reconsidere. Jobs muito longos bloqueiam slots de execução, são difíceis de depurar e geralmente indicam que o trabalho real deveria estar em outra ferramenta (Kubernetes Job, AWS Batch, Airflow). Rundeck brilha em tarefas curtas e frequentes, não em processamento batch de horas.

Runbook automation não é sobre eliminar humanos do loop — é sobre dar a cada humano o poder de operar o sistema com segurança.
